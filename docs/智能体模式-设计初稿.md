## 一、业界调研：主流智能体架构的共同骨架

对 Claude Code、Devin、OpenAI Codex（CLI/云端）、OpenHands（原 OpenDevin）、
Cursor Agent、Manus 等做横向拆解，尽管产品形态各异，核心架构高度趋同：

### 1.1 共同骨架：单线程主循环（Agent Loop）

所有主流智能体的核心都是一个极简循环，而不是复杂的多智能体编排：

```
while (未完成 && 未超限) {
  response = LLM(system prompt + 任务 + 历史事件流 + 工具定义)
  if (response 含工具调用) {
    result = 执行工具(可能先过审批)
    事件流.append(工具调用 + 结果)
  } else {
    完成 / 向用户提问
  }
}
```

- **Claude Code**：官方明确描述为"单线程主循环 + 工具"，复杂性全部下放到工具
  （文件读写、bash、搜索、子任务）而非编排层。
- **OpenHands**：`AgentController` 驱动 `step()` 循环，事件流（EventStream）是
  唯一事实源，所有 Action/Observation 都是事件。
- **Devin**：planner + 执行循环，长任务自动摘要 + 检查点，用户可随时介入。

**结论：不需要多 agent 编排框架。一个循环 + 好工具 + 好上下文管理 = 一线水平。**
这正好和本项目现状吻合：`chat_controller_streaming.dart` 里已经有一个
「LLM → 工具调用 → 结果回填 → 续跑」的循环（最多 N 轮），智能体引擎
本质是把这个循环从「消息树附属品」升级为「独立状态机驱动的长任务」。

### 1.2 六个关键机制（我们要逐个对标）

| 机制 | 业界做法 | 代表 |
|---|---|---|
| **计划/待办** | 显式 todo/plan 工具，模型自己维护任务清单，防长任务跑偏 | Claude Code TodoWrite、Devin planner、Manus 把 todo 重写进上下文尾部对抗"中间遗忘" |
| **上下文管理** | 事件流 + 自动压缩（compaction）：接近窗口上限时把旧事件摘要成一条，工具大输出截断落盘、按需回读 | Claude Code auto-compact、Devin 自动总结、OpenHands condenser |
| **权限分级** | 只读工具直通；写/执行工具过审批；用户可配置白名单（allowlist）；plan-mode 只读规划 | Claude Code permission modes + allowlist、Codex approval modes（suggest/auto-edit/full-auto）|
| **检查点/恢复** | 每步落库，进程死了可恢复；文件改动可回滚 | Devin snapshot、Claude Code checkpoint（v2）、OpenHands 事件流重放 |
| **子任务（subagent）** | 主循环派生一个独立上下文的子循环干专项活（如大范围搜索），只把结论带回主上下文，防止污染 | Claude Code Task/subagent、Devin 子会话 |
| **环境指令/技能** | 从工作区读 AGENTS.md/CLAUDE.md 注入系统提示；技能（skills）按需加载 | Codex AGENTS.md、Claude Code CLAUDE.md + skills |

### 1.3 对移动端的特殊约束（业界没有现成答案，需要自己定）

- **进程生命周期**：手机 App 会被切后台/杀进程，"用户离场也在跑"不能像云端那样
  假设进程常驻 → 必须 **每步落库 + 可恢复** 作为一等公民，前台服务保活为辅。
- **算力在别处**：真正跑命令靠工作区后端（SSH / Termux / PRoot），恰好本项目
  的 `WorkspaceBackend` 抽象已就位——agent 引擎在手机上只做"编排"，副作用
  全部发生在工作区后端里，天然接近"云端沙箱"模型。

## 二、现状盘点：能力层已经齐了八成

设计构想里说"共享能力层，隔离领域层/执行层/存储/任务态 UI"。逐项盘点现状：

| 智能体需要 | 现状 | 复用方式 |
|---|---|---|
| LLM 调用（流式、function calling） | ✅ provider 层 + `chat_controller_streaming` 的流式工具循环 | 抽出/仿写为 agent 引擎的一步（见 §四） |
| 文件读写工具 | ✅ `@aether/file-editor` 15 个工具（读 5 + 写 10），按 workspace root 隔离 | 直接复用 `ToolRoute` 分发 |
| 命令执行 | ✅ `@aether/terminal`（terminal_execute / terminal_session），命令三级风险评级 | 直接复用 |
| 知识库/记忆/网搜 | ✅ `@aether/knowledge`、search_memory、builtin_web_search | 直接复用 |
| HITL 审批 | ✅ `tool_confirmation_service`（60s 超时、5 分钟宽限窗）+ **工具授权白名单**（刚落地，持久化、越界命令硬约束） | 直接复用，agent 侧加"运行级审批策略"（见 §七） |
| 技能 | ✅ skills 商店 + `read_skill` 工具 | 直接复用 |
| 上下文压缩 | ✅ `context_condense_service`（聊天已有压缩） | 改造为事件流 compaction |
| 工作区 | ✅ 文件树/编辑器/多会话终端/SSH/Termux/PRoot/SAF | agent 的"手术台"，只依赖 `WorkspaceBackend` |
| 计划/待办工具 | ❌ 无 | 新增（P0，很小） |
| 任务状态机 + 事件流存储 | ❌ 无 | 新增（P0 核心） |
| 检查点/恢复 | ❌ 无 | 事件流落库天然获得（P0） |
| 子任务 | ❌ 无 | P2 |
| diff/产物面板 | ⭕ 有只读 git diff 染色 | P1 增强 |

**结论：新做的东西 = 领域层（状态机+事件流+存储）+ 执行引擎（把现有工具循环
从消息树里解放出来）+ 任务态 UI + 模式切换壳。工具层几乎零新增。**

## 三、模式切换与持久化（用户设想的入口形态）

### 3.1 形态

- 智能体是**全新的主界面**（`/agent`），与聊天主界面（`/`）**同级**，不是聊天里的一个开关。
- 互切入口：两个主界面的顶栏（或侧栏底部，与「工作区」「翻译」入口同排）放
  模式切换按钮；聊天 → 智能体、智能体 → 聊天。
- **模式持久化**：切换时写 `appSettingsStore`（key: `app_main_mode` = `chat` | `agent`）；
  冷启动时读取该值决定初始主界面（GoRouter `redirect` 或 `initialLocation`）。
  退出前在智能体模式 → 重启进智能体模式，反之亦然。

### 3.2 路由

```
/          聊天主界面（现状不动）
/agent     智能体主界面：任务列表（新）
/agent/task/:taskId   任务详情（新）
```

聊天现有的全部子路由（设置、工作区、笔记……）保持不变；智能体页可直接
push 既有页面（工作区、设置）——设置/工作区本来就是"产品级"页面，两个模式共用。

## 四、领域模型与状态机（`features/agent/`，与 chat 完全隔离）

### 4.1 实体

```dart
/// 一个任务 = 用户的一次委托（对标 Devin session / Claude Code 一次运行）
class AgentTask {
  String id;
  String title;            // 首条指令自动生成，可改
  String workspaceId;      // 绑定的工作区（手术台），可为空=无工作区纯问答
  String modelId;          // 执行模型
  AgentTaskStatus status;  // 状态机，见 4.2
  AgentPermissionMode permissionMode; // 见 §七
  DateTime createdAt, updatedAt;
}

/// 事件流：任务内发生的一切，append-only（对标 OpenHands EventStream）
/// 一条事件 = 一个 sealed 变体：
///   UserMessageEvent      用户指令 / 追加指令 / 审批答复
///   AssistantTextEvent    模型文字（思考/汇报，流式）
///   ToolCallEvent         工具调用（名称+参数+审批状态）
///   ToolResultEvent       工具结果（大输出截断，全文落盘引用）
///   PlanUpdateEvent       计划/待办更新（update_plan 工具产生）
///   CompactionEvent       上下文压缩摘要（替代其覆盖的旧事件参与后续上下文）
///   StatusChangeEvent     状态迁移记录（含原因：出错/用户暂停/超限……）
class AgentEvent { String id; String taskId; int seq; ...payload; }

/// 产物：任务产出的可展示结果（P1）
class AgentArtifact { String id; String taskId; ArtifactKind kind; ... }
// kind: fileDiff / fileSnapshot / commandOutput / summary
```

**不复用 chat 的 `Message/MessageBlock`**：消息树是"对话"模型（分支/重发/多模型），
事件流是"执行审计"模型（append-only、seq 严格递增、可重放）。两者语义不同，
强行复用会把 chat 的复杂度（版本树、块状态机）拖进来。构想文档的隔离结论维持。

### 4.2 任务状态机

```
queued ──▶ running ──▶ done
             │ ▲          
             │ │ resume    
             ▼ │          
   ┌─ waitingApproval（HITL 审批中，60s 不答复自动 paused 而非拒绝*）
   ├─ waitingInput   （模型主动向用户提问）
   ├─ paused         （用户手动暂停 / App 被杀后恢复为 paused）
   ├─ failed         （连续错误超限 / 模型不可用）
   └─ cancelled      （用户终止）
```

\* 与聊天不同：聊天里审批超时=拒绝并继续；agent 长任务里用户可能离场，
超时应**挂起任务**等用户回来，而不是替用户做决定。

### 4.3 存储（新 drift 表，独立于 chat 表）

```
agent_tasks   (id, title, workspace_id, model_id, status, permission_mode, ...)
agent_events  (id, task_id, seq, kind, payload_json, created_at)
              -- 索引 (task_id, seq)；payload 大字段（工具全量输出）超过阈值
              -- 落应用私有目录文件，行内只存引用，防表膨胀
agent_artifacts (P1)
```

每个事件产生即落库 → **检查点免费获得**：App 被杀后任务变 paused，
重进时从事件流重建上下文继续跑（业界"事件流重放"模式）。

## 五、执行引擎（`features/agent/application/agent_engine.dart`）

### 5.1 主循环

```dart
while (task.status == running) {
  context = 组装上下文(系统提示, 事件流);        // §5.2 / §5.3
  response = await llm.stream(context, tools);   // 复用 provider 层
  for (toolCall in response.toolCalls) {
    if (需要审批(toolCall) && !白名单命中) {      // 复用 tool_auth_policy
      task.status = waitingApproval; 等用户;      // UI 推送审批卡片
    }
    result = await 执行(toolCall);               // 复用 ToolRoute 分发
    events.append(ToolCallEvent, ToolResultEvent);
  }
  if (response 无工具调用) {
    是收尾汇报 → done；是提问 → waitingInput;
  }
  安全阀: 轮数/token/连续失败 超限 → paused + 说明;
}
```

与聊天工具循环的三个关键差异：

1. **宿主不同**：不挂在消息树上，挂在 `AgentTask` 事件流上；
2. **审批语义不同**：超时挂起而非拒绝（§4.2）；
3. **无限轮**：聊天循环有固定最大轮数保守值；agent 以预算制
   （token 预算 + 轮数上限 + 连续失败上限）替代，超限挂起让用户续。

实现上**不改动** `chat_controller_streaming.dart`；把工具执行段
（route 分发、审批、取消信号）中可共享的纯逻辑下沉到共享 helper，
聊天与 agent 各自调用——避免"为了复用把两个模式焊死"。

### 5.2 系统提示组装（对标 Claude Code / Codex）

```
[角色与行为规范]   固定：自主多步、先探索后修改、小步验证、何时提问
[计划纪律]         必须用 update_plan 维护待办；长任务每轮开头回顾计划
[工作区上下文]     复用现有 workspaceContext（root、文件树摘要）
[环境指令]         若工作区根有 AGENTS.md，读入注入（对标 Codex/Claude Code）★新增
[技能索引]         复用 skills 目录 + read_skill
[工具清单]         复用 McpSetup 组装（file-editor + terminal + knowledge + 网搜）
```

### 5.3 上下文管理（长任务的命门）

1. **工具大输出截断**：结果超过阈值（如 4k 字符）截断进上下文，全文落盘，
   提示模型可用 `read_file`/分页工具回读（Manus/Claude Code 模式）；
2. **自动 compaction**：事件流 token 估算超过窗口的 ~70% 时，把最老的一段
   事件用辅助模型摘要成一条 `CompactionEvent`（复用/改造
   `context_condense_service`），原事件仍在库里（审计不丢），只是不再进上下文；
3. **计划置尾**：每轮把当前 todo/plan 附在上下文末尾，对抗长上下文"中段遗忘"。

### 5.4 新增的 agent 专属工具（很小，都是纯逻辑）

- `update_plan`：模型维护待办清单（写 `PlanUpdateEvent`，UI 实时渲染）；
- `ask_user`：显式向用户提问并挂起（区别于"文字里顺嘴一问"）；
- `finish_task`：显式收尾 + 结果摘要（驱动状态机到 done）。

### 5.5 生命周期与后台

- P0：**前台可跑 + 杀进程可恢复**。引擎跑在 Dart 侧，App 在前台/最近任务里
  正常执行；进程死亡 → 任务标 paused，重进一键"继续"。
- P1：Android 前台服务（`foregroundServiceType=dataSync`）+ 常驻通知
  展示任务进度，切后台继续跑；网络型工作区（SSH）断连自动重连恢复会话。
- 不做云端执行（超出 App 边界）。

## 六、任务态 UI（`features/agent/presentation/mobile/`）

### 6.1 智能体主界面（任务列表，`/agent`）

- 顶栏：模式切换按钮（回聊天）、新建任务；
- 任务卡：标题、绑定工作区、状态徽标（running 有动效 / waitingApproval 高亮）、
  最近一条事件摘要、耗时/token；
- 新建任务 = 选工作区（或无）+ 选模型 + 输入指令 → 直接进详情页开跑。

### 6.2 任务详情页（`/agent/task/:id`，核心页面）

三段式（对标 Devin 会话页）：

1. **计划区**（可折叠）：当前 todo 清单，完成项打勾——`PlanUpdateEvent` 驱动；
2. **事件流**：按 seq 渲染——模型文字气泡 / 工具卡片（复用现有
   工具块视图：run_command_block、diff 块、terminal_session 块的展示件）/
   审批卡片（批准/拒绝/加入白名单，复用确认服务 UI 语义）/ compaction 分隔线；
3. **输入条**：追加指令（排队为 `UserMessageEvent`，下一轮生效）、
   暂停/继续/终止按钮。

### 6.3 审批体验

- 任务详情内嵌审批卡；App 在后台时发系统通知（P1）；
- 审批卡上提供"本任务内不再询问此工具"（运行级宽限）与
  "永久加入白名单"（写入刚做的 `tool_auth_policy`）两档快捷入口。

## 七、权限与安全（在现有基座上加"运行级策略"）

现有三层不动：只读直通 → 写/执行 HITL → **工具授权白名单**（持久化）→
越出工作区 root 的终端命令**永远强制审批**（硬约束，白名单/宽限均不可覆盖）。

agent 侧新增**每任务的 permission mode**（对标 Codex approval modes）：

| 模式 | 行为 |
|---|---|
| `plan`（只读规划） | 只允许读类工具；写/执行一律不发起（模型被告知先出方案） |
| `ask`（默认） | 现状行为：写/执行过审批，白名单命中免审 |
| `auto`（放手跑） | 工作区 root 内的写/执行全部免审；越界命令仍强制审批 |

`auto` 模式创建任务时需二次确认，并在任务卡上常驻醒目徽标。
所有工具调用无论是否免审都写事件流 → 完整审计。

## 八、分期计划

### P0 —— 能跑通一个真任务（最小闭环）
1. 领域层 + drift 表 + 事件流（§四）；
2. 执行引擎主循环 + update_plan/ask_user/finish_task 工具 + 预算安全阀（§五）；
3. 复用现有工具链与审批白名单，审批超时挂起语义；
4. `/agent` 任务列表 + 任务详情（计划区/事件流/输入条）；
5. 模式切换按钮 + 模式持久化（§三）；
6. 杀进程恢复（事件流重建 + 一键继续）。

### P1 —— 好用
7. 自动 compaction + 工具大输出截断落盘回读；
8. permission mode（plan/ask/auto）；
9. AGENTS.md 环境指令注入；
10. diff/产物面板 + 改动文件清单；
11. Android 前台服务后台跑 + 审批系统通知。

### P2 —— 对标一线
12. 子任务（独立上下文的子循环，结论回传）；
13. 任务模板/定时任务；多任务并行（每任务独立终端会话池已具备）；
14. 检查点回滚（结合工作区 git：任务开始自动记录基线，一键还原）。
