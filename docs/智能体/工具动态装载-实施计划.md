# 工具动态装载（渐进披露）——实施计划

> 状态：PR-A 已实施（机制 + 浏览器试点，见 lib/app/di/
> dynamic_tool_catalog.dart）；本文档同步修订为实施后的事实。
> PR-B 扩围仍为计划。
> 背景：mcp_manage（PR #752）已落地「最小 schema 常驻 + 正文进 skill」的
> 形态；本计划把这套渐进披露推广为通用机制——**工具组默认不进 tools
> 列表，模型 read_skill 读取对应技能后，下一轮才注入该组工具定义**。
> 对标 Claude Code 的 deferred tool loading。

## 1. 目标与非目标

### 目标

- 常驻上下文只保留核心工具（控制类、read_skill、文件/终端等高频组），
  低频大组（首批试点：浏览器 11 个工具）延迟装载。
- 模型保留完整 schema 校验：装载后 tools 列表里是原样定义，
  不走「JSON 字符串转发」的 bridge 路线。
- 激活状态可恢复：任务续跑 / 应用重启后重放，已激活的工具组不丢。

### 非目标

- 不改变审批/权限模型（路由与 HITL 判定不感知装载状态）。
- 不动聊天（chat）侧的工具装配，先只做智能体（agent）侧。
- spawn_subagent、mcp_manage 本来就是单工具+skill，保持现状。

## 2. 现状关键事实（已核实，2026-07）

- tools 列表**每轮请求都重新发送**：`_GatewayAgentLlmClient`
  持有 `List<McpToolDefinition> _definitions`，每次
  `LlmChatRequest(tools: _definitions, ...)`（
  `lib/app/di/agent_runtime_access.dart`）。列表是可变实例，且与
  `forProfile` 装配的 catalog 共享——**中途改动这个 List，下一轮请求
  即生效**，这是本方案可行的根基。
- catalog 在 `AgentRuntime.forProfile` → `_catalogFor` 一次性构建：
  `definitions`（发给模型）与 `routes`（执行分发 Map）同源。
- `read_skill` 经 `SkillReadToolRoute` 在 `ChatToolExecutor` 内执行
  （`lib/shared/mcp_tools/skill_read_tool.dart` 的 `executeReadSkill`）。
- 浏览器组当前在 `_catalogFor` 里按 `AgentToolGroup` 全量注入
  `kBuiltinMcpTools['@aether/browser']` 的 11 个定义。
- 内置技能「内置浏览器」（`builtin-browser`，
  `lib/shared/config/builtin_skills/browser_skill.dart`）已包含完整用法。
- 系统提示已注入「可用技能」清单（read_skill 按名称读取）。

## 3. 设计

### 3.1 核心抽象：DynamicToolCatalog

已落地在 `lib/app/di/dynamic_tool_catalog.dart`（agent↔chat 架构
隔离——组装在 app/di 最稳）。无可变激活状态，激活集每轮从事件流
重算后传入：

```dart
class DynamicToolCatalog {
  final List<McpToolDefinition> resident;      // 常驻定义
  final Map<String, List<McpToolDefinition>> deferred; // skillId → 定义组
  final Map<String, ToolRoute> routes;         // 全量路由（不裁剪！）

  List<McpToolDefinition> definitionsFor(Set<String> activatedSkillIds) =>
      [...resident, for (final id in activatedSkillIds) ...?deferred[id]];
}
```

要点：

- **routes 永远全量**。routes 不发给模型，只用于执行分发与重放
  （`_replayMessages` 里历史工具结果要能找到路由）；只裁剪 definitions。
- `_GatewayAgentLlmClient` 改为持 `DynamicToolCatalog`，每轮
  `tools: catalog.definitionsFor(已激活集)`。
- 审批门 `_PolicyApprovalGate` 继续用全量 routes，零改动。

### 3.2 激活判定：每轮扫原始事件流（实施时的方案 C，优于原 A/B）

实施时未采用「 runTool 执行处挂钩 + catalog 可变状态」，而是
**每轮 completeTurn 对原始事件流无状态重算**：扫描成功的
read_skill 工具事件，解析技能后查 `kSkillToolBindings` 得到已激活
集合（`activatedSkillIdsFromEvents`）。优点：活跃激活、续跑/重启
恢复、agentContextBreakdown 三个消费点共用同一代码路径，无可变
状态可错位。对比 CC：同款思路（CC 的 `extractDiscoveredToolNames`
也是每请求扫消息历史里的 tool_reference）。

名称匹配复用 `matchSkillByName`（从 `executeReadSkill` 抽出的
「精确 → 忽略大小写 → 子串」三段式共用函数），避免读到了技能但
没激活工具的错位。激活天然幂等（集合语义）：重复 read_skill
无副作用，不报错——对标 CC 对已装载工具 select 也返回成功，
避免模型重试抖动。

### 3.3 激活状态恢复（重启/续跑）与压缩安全

不加新存储：事件流 append-only，天然是激活记录，3.2 的每轮重算
已含恢复语义。

**压缩安全（实施时新增的硬约束）**：激活扫描必须基于**原始
事件流**，不能基于 fold/microcompact 后的视图——压缩摘要会吞掉
read_skill 工具事件，导致压缩后激活状态丢失（CC 专门用
`preCompactDiscoveredTools` 快照防的同一问题）。当前实现扫
`context.events`（原始流），fold 只发生在 `_replayMessages` 内部，
天然满足；将来若改为传入已折叠视图，需先把激活集快照进
压缩事件。

### 3.4 模型引导

- 常驻上下文的「可用技能」清单里，对绑定了工具组的技能加一句：
  「读取本技能后，对应工具将在下一轮可用」（已实施，见
  `_skillsSection`）。
- 被延迟的组在常驻里**不留占位定义**（CC 同款）；模型看到任务需要
  浏览器 → 从技能清单知道有「内置浏览器」→ read_skill → 下轮工具出现。
- 技能正文开头补一行：「本技能绑定的工具已随本次读取装载，
  下一轮起可直接调用；重复读取无副作用」（已实施，浏览器技能
  version 1.6.0 → 1.7.0）。

### 3.5 绑定关系声明

在内置技能定义上加可选字段（`Skill` 已是 freezed，加字段跑
build_runner），或先用 app/di 里的常量 Map 起步（改动最小，推荐先这样）：

```dart
const kSkillToolBindings = {
  'builtin-browser': '@aether/browser', // skillId → builtin server name
};
```

### 3.6 其他必须同步的消费点（实施时核实的遗漏，已修）

- **`agentContextBreakdown`**（工作台「上下文」tab）：原先用
  `catalog.definitions` 估算 token，必须同样按事件流重建激活集用
  `definitionsFor(...)`，否则显示与真实请求不符。
- **系统提示的 `_definitions` 依赖**：环境上下文的「可用工具」
  清单改为列当轮实际发送的定义；read_skill / spawn_subagent 的
  存在性检查改查常驻集（两者恒常驻，语义不变）。
- **技能库加载失败时 fail-open**：扫描无法解析技能时视为全部激活，
  宁可多发定义不能让已激活工具在续跑时消失。

2. **PR-B 扩围**（视效果，未实施）：评估把 knowledge 组（kb_*）、
   web_search 等低频组也 deferred；子代理与 mcp_manage 维持单工具常驻。
   可选增强：对标 CC `ENABLE_TOOL_SEARCH=auto:N` 的阈值自动开关
   （延迟组定义超过上下文窗口 N% 才启用渐进披露）。

## 5. 风险与对策

- **模型不读技能就想调工具** → routes 全量，幻觉调用延迟工具仍能
  正常执行（无害宽容，CC 对已装载工具 select 也返回成功同款语义，
  避免重试抖动）；真正未知的工具错误信息已补「先 read_skill 读取
  对应技能」提示。
- **子代理**：子代理有独立 catalog，浏览器组直接常驻（子代理上下文
  短，无需渐进披露；已实施，以 `enableSubagents=false` 区分）。
- **Ask/Plan 只读模式**：deferred 组在 `_catalogFor` 构建时就按
  readOnly 规则过滤（与常驻组同一把尺），激活后注入的只有只读子集。
- **token 收益量化**：浏览器 11 个定义 JSON 的字符量写进 PR 描述
  作为依据。
