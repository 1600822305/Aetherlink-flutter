## 1. 现状（我们的实现）

**核心设计**：事件流是审计事实源**永不改写**，压缩只作用于「进模型上下文的视图」。

| 组件 | 位置 | 说明 |
|---|---|---|
| 折叠算法 | `lib/features/agent/application/engine/agent_compaction.dart` | `foldCompactedEvents`（纯函数，引擎与重放侧共用）、`contextCharsOf` / `totalContextChars`（字符估算）、`selectCompactionPrefix`（选覆盖前缀，保底最近 8 条） |
| 触发与执行 | `agent_engine.dart` 的 `_maybeCompact` | 每轮安全点检查；失败不阻断（下轮再试）+ 可见提示；已接 `onPreCompact` / `onPostCompact` hook 回调 |
| 阈值 | `agent_budget.dart` | `compactionTriggerChars = 120000`、`compactionKeepChars = 40000`（固定字符数，与模型无关） |
| 摘要生成 | `agent_runtime_access.dart` 的 `summarizeForCompaction` | 一次 LLM 调用，中文单段系统提示词（保留目标/动作/路径/待办） |
| 落库 | `CompactionEvent(coveredCount, summary)` | 摘要作为事件追加，可被更晚的压缩再次覆盖 |

**优势（保持，勿破坏）**：
- 事件流不可变、可审计；CC 直接改消息数组，我们不学。
- 折叠是纯函数 + 单测（`engine/agent_engine_test.dart` 的 compaction 组），行为可预测。
- 失败不阻断任务。

## 3. 升级任务（按性价比排序，逐项独立提交）

### ⬜ ① microcompact：工具输出占位清除（不调 LLM，性价比最高）

**目标**：上下文超「micro 阈值」（低于 LLM 压缩阈值）时，把**较旧的**大体积工具输出
（`ToolCallEvent.resultDetail`）在**进上下文的视图里**替换为占位符
`[旧工具输出已清除]`，近期 N 条工具调用不动。不落新事件、不调 LLM、零成本。

**设计要点**：
- 遵守事件流不可变：不改 `ToolCallEvent` 本体，在折叠视图层做替换——
  `agent_compaction.dart` 新增纯函数（如 `microCompactEntries`），引擎构建消息前
  与重放侧（`_replayMessages`）共用，与 `foldCompactedEvents` 同款共享模式。
- 可清除范围对标 CC：只清「可重取」的工具输出（终端执行 / 文件读 / 搜索类），
  文件编辑 diff 等语义结果保留摘要行（`resultSummary`）；工具白名单做成常量便于调整。
- 需要一个「已清除到哪」的锚点：建议用确定性规则（视图内倒数第 K 条之前 + 单条超
  N 字符才清），纯函数无状态，两侧天然一致，避免落库新事件。
- 触发阈值：`agent_budget.dart` 加 `microCompactTriggerChars`（建议 80k，
  低于 LLM 压缩的 120k，先 micro 后 LLM 的两级降压）。

**验收**：纯函数单测（清除规则/保底近期条数/两侧一致）；超阈值时进上下文字符量下降；
`CompactionEvent` 摘要不受影响；analyzer + agent 单测全绿。

### ⬜ ② 摘要提示词升级为 CC 式结构化要求

**目标**：`summarizeForCompaction` 的系统提示词升级：要求模型先做 `<analysis>`
分析（对话里每段的意图/关键信息），再输出结构化摘要（沿用中文）：
1. 用户目标与约束（原话要点） 2. 关键概念/架构 3. 涉及文件与代码片段（保留路径与
签名） 4. 报错与修复方式 5. 已解决的问题 6. 待办事项 7. 当前正在做的事
8. 下一步（含用户最近的明确指示）。`<analysis>` 块在落库前剥离，只存摘要正文。

**验收**：提示词落在 `agent_runtime_access.dart`（或抽到 domain 常量便于单测）；
剥离 `<analysis>` 有单测；长任务手测摘要质量明显改善。

### ⬜ ③ 触发阈值改按模型上下文窗口 + 真实 usage

**目标**：不再固定 120k 字符：
- `Usage`（`features/chat/domain/entities/usage.dart`）已有真实 `promptTokens`——
  引擎在每轮 done chunk 处记录最近一次 usage，作为「当前上下文 token」的第一依据；
  拿不到 usage（部分供应商不回）时回退现有字符估算。
- 模型上下文窗口：`shared/domain/model.dart` 若无窗口字段则补（按模型 id 内置常见
  映射 + 用户可覆盖），有效窗口 = 窗口 − min(模型最大输出, 20k)（对标 CC 摘要预留）。
- 触发比例建议 92%（CC 同量级）；`compactionKeepChars` 同步改为比例制。

**验收**：有 usage 时按 token 触发、无 usage 回退字符；阈值计算纯函数 + 单测；
不同窗口模型（8k/128k/200k）行为正确。

### ⬜ ④ 压缩预警 + 熔断

**目标**：
- 预警：距触发阈值 ≤10% 时落一条可见状态事件（一次任务只提示一次），并触发
  `notification` hook（type 新增 `compactWarning`，复用 ⑦-2 通道）。
- 熔断：连续 N 次（建议 3）压缩失败后本任务内停止再尝试（避免每轮白调一次 LLM），
  落一条可见提示；成功一次即重置计数。

**验收**：预警只出一次；熔断后不再调 summarize；单测覆盖计数/重置。

### ⬜ ⑤ 手动压缩入口（可选，UI 项）

**目标**：任务详情页加「立即压缩」操作（对标 `/compact`）：暂停安全点触发一次
`_maybeCompact` 的强制版（忽略触发阈值，仍走 keep 规则）。运行中任务在下一个
安全点执行。

**验收**：手动触发落 `CompactionEvent`；折叠视图变小；对已完成任务禁用或提示。

### 不做（明确排除）

- **按时间的 microcompact / prompt cache 对齐**：绑定 Anthropic 缓存计费语义，
  本项目多供应商形态不契合。
- **session memory 压缩**：CC 实验特性，不稳定。
- **改写历史消息数组**：违背本项目「事件流不可变」核心设计，永不采纳。
