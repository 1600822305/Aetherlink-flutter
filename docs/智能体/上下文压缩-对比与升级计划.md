# 上下文压缩（Compaction）：对比与升级计划

> **给 AI 助手 / 未来会话的说明**：这是上下文压缩升级的任务台账，工作方式与

## 1. 现状（我们的实现）

**核心设计**：事件流是审计事实源**永不改写**，压缩只作用于「进模型上下文的视图」。

| 组件 | 位置 | 说明 |
|---|---|---|
| 折叠算法 | `lib/features/agent/application/engine/agent_compaction.dart` | `foldCompactedEvents`（纯函数，引擎与重放侧共用）、`contextCharsOf` / `totalContextChars`（字符估算）、`selectCompactionPrefix`（选覆盖前缀，保底最近 8 条） |
| 触发与执行 | `agent_engine.dart` 的 `_maybeCompact` | 每轮安全点检查；失败不阻断（下轮再试）+ 可见提示；已接 `onPreCompact` / `onPostCompact` hook 回调 |
| 阈值 | `agent_budget.dart` | `compactionTriggerChars = 120000`、`compactionKeepChars = 40000`（固定字符数，与模型无关） |
| 摘要生成 | `agent_runtime_access.dart` 的 `summarizeForCompaction` | 一次 LLM 调用，中文单段系统提示词（保留目标/动作/路径/待办） |
| 落库 | `CompactionEvent(coveredCount, summary)` | 摘要作为事件追加，可被更晚的压缩再次覆盖 |

**优势（保持，勿破坏）**：
- 事件流不可变、可审计；CC 直接改消息数组，我们不学。
- 折叠是纯函数 + 单测（`engine/agent_engine_test.dart` 的 compaction 组），行为可预测。
- 失败不阻断任务。

## 3. 升级任务（按性价比排序，逐项独立提交）

### ✅ ① microcompact：工具输出占位清除（不调 LLM，性价比最高）

**目标**：上下文超「micro 阈值」（低于 LLM 压缩阈值）时，把**较旧的**大体积工具输出
（`ToolCallEvent.resultDetail`）在**进上下文的视图里**替换为占位符
`[旧工具输出已清除]`，近期 N 条工具调用不动。不落新事件、不调 LLM、零成本。

**设计要点**：
- 遵守事件流不可变：不改 `ToolCallEvent` 本体，在折叠视图层做替换——
  `agent_compaction.dart` 新增纯函数（如 `microCompactEntries`），引擎构建消息前
  与重放侧（`_replayMessages`）共用，与 `foldCompactedEvents` 同款共享模式。
- 可清除范围对标 CC：只清「可重取」的工具输出（终端执行 / 文件读 / 搜索类），
  文件编辑 diff 等语义结果保留摘要行（`resultSummary`）；工具白名单做成常量便于调整。
- 需要一个「已清除到哪」的锚点：建议用确定性规则（视图内倒数第 K 条之前 + 单条超
  N 字符才清），纯函数无状态，两侧天然一致，避免落库新事件。
- 触发阈值：`agent_budget.dart` 加 `microCompactTriggerChars`（建议 80k，
  低于 LLM 压缩的 120k，先 micro 后 LLM 的两级降压）。

**验收**：纯函数单测（清除规则/保底近期条数/两侧一致）；超阈值时进上下文字符量下降；
`CompactionEvent` 摘要不受影响；analyzer + agent 单测全绿。

✅ **已完成（2026-07-19）**：新模块 `application/engine/agent_microcompact.dart`
（与 agent_compaction 同层，不堆进既有文件）：`microCompactEntries` 确定性
纯函数（从最旧清白名单内、输出 ≥2000 字符的工具 resultDetail → 占位符，
直到降到阈值内或触达最近 5 条工具调用的保护窗口；未触发时零开销原样返回）。
白名单只含可重取输出（终端/文件读/搜索/网络/知识库检索）；CC 的
COMPACTABLE_TOOLS 连 edit/write 也清，本项目刻意更保守——编辑 diff 是语义
结果，不清（2026-07-19 复查时对照 CC 源码修正了此处此前的表述）。
引擎 `_maybeCompact` 与重放侧 `_replayMessages` 都改为「先折叠再 micro」同款
视图；阈值 `kMicroCompactTriggerChars = 80000`（budget 新增同名字段默认引用），
与 LLM 压缩 120k 构成两级降压。coveredCount 语义不变（micro 不改列表长度），
事件流本体永不改写。单测 `engine/agent_microcompact_test.dart` 6 个用例；
agent 全量 165 个单测通过。

### ✅ ② 摘要提示词升级为 CC 式结构化要求

**目标**：`summarizeForCompaction` 的系统提示词升级：要求模型先做 `<analysis>`
分析（对话里每段的意图/关键信息），再输出结构化摘要（沿用中文）：
1. 用户目标与约束（原话要点） 2. 关键概念/架构 3. 涉及文件与代码片段（保留路径与
签名） 4. 报错与修复方式 5. 已解决的问题 6. 待办事项 7. 当前正在做的事
8. 下一步（含用户最近的明确指示）。`<analysis>` 块在落库前剥离，只存摘要正文。

**验收**：提示词落在 `agent_runtime_access.dart`（或抽到 domain 常量便于单测）；
剥离 `<analysis>` 有单测；长任务手测摘要质量明显改善。

✅ **已完成（2026-07-19）**：新模块 `application/engine/agent_compaction_prompt.dart`
（提示词常量 + 解析纯函数，独立于执行层）：`kCompactionSummarySystemPrompt`
为 CC BASE_COMPACT_PROMPT 中文对齐版（禁用工具前言 + <analysis> 逐段梳理 +
<summary> 9 小节：请求意图/关键概念/文件与代码/报错与修复/已解决/用户消息
记录/待办/当前工作/下一步）；`extractCompactionSummary` 剥离 <analysis>
草稿块、解包 <summary>，模型未按格式输出时原样返回全文（格式失败不丢摘要）。
`summarizeForCompaction` 改用两者。单测 `engine/agent_compaction_prompt_test.dart`
7 个用例。

### ✅ ③ 触发阈值改按模型上下文窗口 + 真实 usage

**目标**：不再固定 120k 字符：
- `Usage`（`features/chat/domain/entities/usage.dart`）已有真实 `promptTokens`——
  引擎在每轮 done chunk 处记录最近一次 usage，作为「当前上下文 token」的第一依据；
  拿不到 usage（部分供应商不回）时回退现有字符估算。
- 模型上下文窗口：`shared/domain/model.dart` 若无窗口字段则补（按模型 id 内置常见
  映射 + 用户可覆盖），有效窗口 = 窗口 − min(模型最大输出, 20k)（对标 CC 摘要预留）。
- 触发比例建议 92%（CC 同量级）；`compactionKeepChars` 同步改为比例制。

**验收**：有 usage 时按 token 触发、无 usage 回退字符；阈值计算纯函数 + 单测；
不同窗口模型（8k/128k/200k）行为正确。

✅ **已完成（2026-07-19）**：新模块 `application/engine/agent_compaction_trigger.dart`
（纯函数）：有效窗口 = 窗口 − min(20k 摘要预留, 窗口/4)，触发阈值 = 有效窗口
× 92%（CC 同量级）；`shouldTriggerCompaction` 在 usage（`task.contextTokens`，
已有真实 promptTokens 回填）与窗口都已知时走 token 判定，否则回退现有字符
估算。窗口来源：`AgentBudget.contextLimitTokens` 新字段（默认 0 = 未知），主任务
与子代理构造处从设置页的上下文长度（`AgentUiSettings.contextLimit`，默认
128k）传入。未引入按模型 id 的窗口映射（设置页已可按模型自行配置，避免维护
内置表）；`compactionKeepChars` 仍为字符制（前缀选择本就按字符估算）。单测
`engine/agent_compaction_trigger_test.dart` 7 个用例；agent 全量 179 个单测通过。

### ✅ ④ 压缩预警 + 熔断

**目标**：
- 预警：距触发阈值 ≤10% 时落一条可见状态事件（一次任务只提示一次），并触发
  `notification` hook（type 新增 `compactWarning`，复用 ⑦-2 通道）。
- 熔断：连续 N 次（建议 3）压缩失败后本任务内停止再尝试（避免每轮白调一次 LLM），
  落一条可见提示；成功一次即重置计数。

**验收**：预警只出一次；熔断后不再调 summarize；单测覆盖计数/重置。

✅ **已完成（2026-07-19）**：新模块 `application/engine/agent_compaction_guard.dart`
（纯逻辑）：`isNearCompactionThreshold` 达触发阈值 90% 且未超阈值时预警，
与触发判定同款双路（token 优先、字符回退）；`CompactionCircuitBreaker`
连续 3 次失败熔断（recordFailure 返回 justOpened 便于只提示一次），成功即重置。
引擎接线：预警一次性落可见状态行 + notification hook（type=compactWarning，
复用 ⑦-2 通道）；熔断后本次运行内 `_maybeCompact` 直接跳过（不再调
summarize），熔断时落一条可见提示，续跑（新引擎实例）恢复。单测
`engine/agent_compaction_guard_test.dart` 5 个用例；agent 全量 184 个单测通过。

### ✅ ⑤ 手动压缩入口（UI 项）

**目标**：任务详情页加「立即压缩」操作（对标 `/compact`）：暂停安全点触发一次
`_maybeCompact` 的强制版（忽略触发阈值，仍走 keep 规则）。运行中任务在下一个
安全点执行。

**验收**：手动触发落 `CompactionEvent`；折叠视图变小；对已完成任务禁用或提示。

✅ **已完成（2026-07-19）**：新模块 `application/engine/agent_manual_compaction.dart`
（引擎与 runner 共享）：`runManualCompaction` 强制压缩一次——忽略触发阈值，
仍走既有的折叠 + microcompact 视图与 keep 前缀选择，落 `CompactionEvent`；
内容太少时返回 NothingToCover。接线：引擎新增 `manualCompactSignal`（调用即
消费，对标 hookStopSignal 形态）——运行中任务在下一个安全点强制压缩（跳过
阈值/预警/熔断）；runner `compactNow` 对运行中任务挂信号，空闲任务（暂停/
完成/失败）直接经 DI runtime 的 llm 执行。UI：智能体设置 tab「上下文」组新增
「立即压缩」行（入口形态对齐聊天的「压缩上下文」：确认弹窗 + toast 反馈；
但语义不同——事件流原文保留可审计，故用轻量确认而非不可恢复警告；代码不跨
feature 共享，守住 agent/chat 不互 import 边界）；草稿话题不展示。单测
`engine/agent_manual_compaction_test.dart` 2 个用例；agent 全量 186 个单测通过。

### 不做（明确排除）

- **按时间的 microcompact / prompt cache 对齐**：绑定 Anthropic 缓存计费语义，
  本项目多供应商形态不契合。
- **session memory 压缩**：CC 实验特性，不稳定。
- **改写历史消息数组**：违背本项目「事件流不可变」核心设计，永不采纳。
