## 一、一线产品循环架构拆解（专业视角）

### 1.1 Claude Code —— "单循环 + 好工具"的极简派

```
主循环（单线程）:
  loop {
    response = LLM(system + 对话历史 + tools)
    if (无 tool_use) → 输出文本，回到用户       // 循环出口只有这一个
    for (tool_use in response) {
      permission = 检查(permission mode + allowlist + hooks)
      if (需要审批) → 挂起等用户（拒绝则把拒绝原因作为 tool_result 回填）
      result = 执行工具
      history.append(tool_use, tool_result)     // 结果永远回填，失败也回填
    }
  }
```

关键设计：
- **复杂性全部下放到工具层**，编排层薄到极致。没有 planner 进程、没有
  多 agent 图，靠 TodoWrite 工具让模型自己维护计划。
- **子任务（Task/subagent）也是一个工具**：主循环调 Task 工具 → 开一个
  全新上下文的子循环 → 只把最终结论作为 tool_result 带回。上下文隔离
  是靠"工具边界"实现的，不是靠编排框架。
- **审批即数据**：拒绝不是异常，而是一条 tool_result（"用户拒绝了此
  操作，原因：…"），模型据此改道。循环本身对审批无感知。
- **auto-compact**：上下文接近窗口上限时自动把旧历史摘要重写，对循环
  透明。
- 出口纪律：模型不调用工具 = 这轮任务收尾（回答/提问），没有显式
  finish 工具（靠 prompt 纪律约束）。

### 1.2 OpenAI Codex（CLI）—— 审批模式作为一等公民

```
turn 循环:
  loop {
    response = LLM(instructions + AGENTS.md + 历史 + tools)
    if (无 tool call) → break
    for (call in response) {
      decision = approval_policy(call)   // suggest / auto-edit / full-auto
      switch (decision) {
        allow   → 沙箱内执行
        ask     → 弹审批，拒绝则回填拒绝并继续
        forbid  → 直接回填"策略禁止"
      }
    }
  }
```

关键设计：
- **审批策略是显式的三档模式**（suggest / auto-edit / full-auto），
  用户一开始就选好"授权级别"，而不是逐条决定；逐条审批只是
  suggest 档的行为。
- **沙箱是执行的默认前提**（seatbelt/landlock），审批档位实际上是在
  调"沙箱逃逸"的门槛。执行环境安全兜底 + 策略放行，两层。
- **AGENTS.md 注入**：项目级指令作为系统提示的一节，权威低于内置
  指令——这直接影响循环行为（如"跑测试再提交"）。

### 1.3 Devin —— 计划层 + 检查点的"长任务"派

- 循环上方有一个**显式计划层**：任务开始先产出计划（用户可改），
  执行中计划状态实时更新，计划是用户介入的主要界面。
- **每步落检查点**：环境快照 + 事件流，浏览器/终端/编辑器状态都可恢复；
  用户可以随时打断、改指令，循环从最近检查点续跑。
- **消息可插队**：用户消息进入队列，循环在下一个安全点消费（不打断
  正在执行的工具），这就是我们事件流里 `queued` 字段的由来。
- **长任务自动摘要**：上下文接近上限自动 compaction，审计流不丢。

### 1.4 OpenHands（原 OpenDevin）—— 事件流即架构

```
AgentController.step():
  state = 事件流重放()                 // 事件流是唯一事实源
  action = agent.step(state)           // LLM 决策 → Action
  observation = runtime.run(action)    // 执行 → Observation
  event_stream.append(action, observation)
```

关键设计：
- **一切皆事件**：用户消息、模型动作、执行结果、状态变更全部是
  append-only 事件；控制器状态可以从事件流完全重放出来。
  这是四家里**与我们已落库的 `agent_events` 模型最同构的**。
- condenser（压缩器）作为事件流的视图层：进上下文的是"压缩视图"，
  库里的原始事件永不修改——审计与上下文管理解耦。

### 1.5 共性总结（= 业界正确答案）

| 共性 | 四家一致的做法 |
|---|---|
| 循环形态 | 单线程 while 循环，无多 agent 编排 |
| 唯一事实源 | 事件流 / 对话历史，append-only |
| 审批 | 是数据不是异常：拒绝回填给模型，循环不中断 |
| 计划 | 模型用工具自维护，置于上下文尾部对抗遗忘 |
| 上下文 | 自动 compaction + 大输出截断落盘按需回读 |
| 恢复 | 每步落库/检查点，重放即恢复 |
| 出口 | "无工具调用"或显式 finish 工具 |

## 二、三视角分析：我们该怎么设计

### 2.1 用户视角（手机上的个人用户）

用户在移动端最在乎的四件事，对循环设计的直接要求：

1. **"它在干嘛"随时可见** → 循环每产生一个事件立即落库 + 推流，
   UI watch 数据库即得实时时间线（我们已建好这条管道）。禁止
   "循环跑完一轮才批量写事件"。
2. **随时能打断、能插话** → 用户消息必须能在循环执行中投递
   （Devin 式插队）：写一条 `queued=true` 的 UserMessageEvent，
   循环在每轮 LLM 调用前消费队列；「停止」按钮 = 协作式取消信号，
   在工具边界生效（不撕裂半个工具调用）。
3. **杀进程不丢活** → 移动端进程随时会死。循环的每一步效应
   （事件、状态、计划）先落库再继续；重启后任务标 paused，
   一键"继续" = 事件流重放出上下文接着跑。这是移动端相对
   云端产品必须**更强**的地方，不是可选项。
4. **审批不烦人** → 只读直通、白名单直通、其余弹卡；审批卡挂起
   任务而非超时拒绝（用户可能锁屏了）。默认档位求稳，白名单
   让熟练用户逐步"放权"。

### 2.2 企业级视角（审计 / 安全 / 成本 / 可靠性）

1. **审计完整性**：事件流 append-only、seq 严格递增、原始事件
   永不改写（compaction 只是视图）——"这个 agent 当时为什么删了
   那个文件"必须能从库里完整回答。我们的 `agent_events` 表已满足，
   循环实现时的红线是：**任何路径都不得跳过事件落库直接产生副作用**。
2. **权限最小化 + 硬约束**：审批策略分层——工具风险分级（已有命令
   三级评级）→ 运行级策略（本次任务的档位）→ 白名单（持久授权）→
   越界硬约束（工作区 root 之外一律禁止，不可被提示词说服）。
   策略判定本身也要出事件（谁批的、命中哪条白名单）。
3. **成本护栏**：无限轮循环必须配预算制——token 预算、轮数上限、
   连续失败上限、单工具超时。超限不是 fail 而是 paused + 说明，
   用户续费/续跑。每轮记录 token 用量进任务统计（已有字段）。
4. **可靠性语义**：崩溃恢复要求循环步进是**幂等可重入**的：
   恢复时以"最后一条完整事件"为准，半途的工具调用按失败回填。
   事件 id 天然做幂等键（我们的 upsertEvents 已按 id 覆盖）。

### 2.3 专家视角（架构可演化性）

1. **循环内核要纯**：`AgentEngine` 只做编排（组上下文 → 调 LLM →
   分发工具 → 落事件 → 判状态），不直接依赖任何具体工具/UI/网络实现。
   依赖全部抽象成接口：`AgentLlmClient`（LLM 流式调用）、
   `AgentToolExecutor`（工具分发）、`ApprovalGate`（审批）、
   `EventSink`（落库+推流）。这样 P2 加子任务/换模型/换执行后端
   都不动内核。
2. **不复用 chat 循环的代码，只复用能力**：`chat_controller_streaming`
   的循环焊死在消息树上，改造成本 > 重写成本。正确做法（设计初稿
   已拍板）：把可共享的纯逻辑（ToolRoute 分发、审批服务、白名单）
   下沉共享层，agent 引擎另起。agent↔chat import 边界继续成立。
3. **状态机显式化**：running / waitingApproval / waitingInput /
   paused / done / failed / cancelled 的每个迁移都由事件驱动
   （StatusChangeEvent），不允许"内存里改了状态但事件流没记录"。
4. **子任务留口不实现**：Claude Code 证明 subagent = 一个工具 +
   一个新事件流上下文。我们的 Task→Event 模型天然支持
   （子任务 = 另一个 AgentTask，parent 字段 P2 再加），
   现在不做，但循环内核不要写死"单任务"假设。

## 三、我们的循环设计（细化版，可直接落码）

### 3.1 内核接口（features/agent/application/engine/）

```dart
/// 循环内核：纯编排，所有依赖注入。
class AgentEngine {
  AgentEngine({
    required AgentLlmClient llm,        // LLM 流式调用（复用 provider 层，经 di seam）
    required AgentToolExecutor tools,   // 工具分发（复用 ToolRoute，经 di seam）
    required ApprovalGate approval,     // 审批（复用 tool_confirmation + 白名单）
    required AgentEventStore store,     // 事件落库（AgentDao 之上的薄封装）
    required AgentBudget budget,        // token/轮数/连续失败预算
  });

  Future<void> run(AgentTask task, CancellationToken cancel);
}
```

### 3.2 主循环（伪码，含移动端细节）

```dart
Future<void> run(task, cancel) async {
  while (true) {
    // ① 安全点：消费插队的用户消息（queued → 正式进上下文）
    await store.consumeQueuedUserMessages(task.id);

    // ② 预算检查（token / 轮数 / 连续失败）
    if (budget.exceeded) return pause(task, 原因说明);
    if (cancel.requested) return cancelled(task);

    // ③ 组上下文：系统提示五层（初稿 §5.2）+ 事件流压缩视图（§5.3）
    //    + 计划置尾
    final context = buildContext(task);

    // ④ LLM 流式调用：AssistantTextEvent 以 streaming=true 先落一条，
    //    增量原位覆盖（id 不变 upsert），流结束置 streaming=false
    final response = await llm.stream(context, toolsFor(task.profile));

    // ⑤ 无工具调用 → 出口判定
    if (response.toolCalls.isEmpty) {
      // finish_task 已调用 → done；ask_user → waitingInput；
      // 否则视为收尾汇报 → done（prompt 纪律 + 显式工具双保险）
      return finishOrAsk(task, response);
    }

    // ⑥ 逐个执行工具调用
    for (final call in response.toolCalls) {
      // 先落 ToolCallEvent(state=running / waitingApproval)
      final event = await store.appendToolCall(task.id, call);

      final decision = await approval.decide(call, task);   // 三层策略
      if (decision.needsUser) {
        await transition(task, waitingApproval);            // 出状态事件
        final verdict = await decision.wait(cancel);        // 挂起（无超时拒绝，超时=保持挂起）
        if (verdict.denied) {
          await store.updateToolCall(event, state: denied,
              result: '用户拒绝：${verdict.reason}');        // 拒绝是数据，回填继续
          continue;
        }
        await transition(task, running);
      }

      final result = await tools.execute(call, cancel)      // 单工具超时
          .timeout(kToolTimeout);
      // 大输出：截断进上下文，全文落盘留引用（初稿 §5.3）
      await store.updateToolCall(event,
          state: result.ok ? success : failure,
          result: truncateAndSpill(result));
      budget.recordToolResult(result);
    }

    // ⑦ 自动 compaction 检查（超窗口 ~70% → 摘要出 CompactionEvent）
    await maybeCompact(task);
  }
}
```

### 3.3 关键决策点（待你拍板）

| # | 决策点 | 建议 | 理由 |
|---|---|---|---|
| L1 | 循环出口 | 显式 `finish_task`/`ask_user` 工具为主，"无工具调用"兜底判收尾 | 纯靠 prompt 纪律（Claude Code 式）在小模型上不稳；显式工具让状态机确定 |
| L2 | 审批语义 | 拒绝回填继续跑（不终止任务）；挂起无超时（初稿 §4.2 已拍板超时挂起） | 四家一致：拒绝是数据；移动端用户可能锁屏，超时拒绝会毁任务 |
| L3 | 用户插话 | queued 事件 + 每轮 LLM 调用前消费（Devin 式安全点插队） | 事件模型已预留 queued 字段 |
| L4 | 取消 | 协作式 CancellationToken，工具边界生效；执行中的终端命令发中断 | 硬杀会留半个事件，破坏可重放性 |
| L5 | 预算默认值 | 轮数 50 / 连续失败 5 / 单工具 5min，token 预算按模型窗口推导，超限 paused | 企业级成本护栏；数值后续设置页可调（骨架已留） |
| L6 | 流式文本落库 | AssistantTextEvent 原位 upsert（id 不变，增量覆盖），不逐 chunk 追加新事件 | 事件表不膨胀；watch 管道天然拿到增量 |
| L7 | 恢复语义 | 重启后 running 任务标 paused；「继续」= 半途工具按失败回填 + 重放上下文续跑 | 幂等可重入，移动端一等公民 |
| L8 | 子任务 | P0 不做，内核不写死单任务假设（执行器/事件流均以 taskId 为轴） | Claude Code 证明后补成本低 |

### 3.4 分步落地顺序（P0 内）

1. **engine 骨架 + 接口**：AgentEngine + 四个接口 + CancellationToken，
   假 LLM/假工具先把状态机、事件落库、恢复语义跑通（可单测）。
2. **接真 LLM**：经 app/di seam 复用 provider 层流式调用，
   系统提示五层组装（初稿 §5.2）。
3. **接真工具**：ToolRoute 分发下沉共享 helper，`update_plan` /
   `ask_user` / `finish_task` 三个新工具。
4. **接审批**：三层策略 + 审批卡接真（UI 骨架已在）。
5. **compaction + 预算 + 恢复**收尾。

每步独立可验证、独立推 main，UI 全程不用大动（事件流管道已通）。
