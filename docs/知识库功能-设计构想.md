# 知识库（Knowledge Base）功能设计构想

> 状态：**构想 / 设计草案**，尚未落地。
> 目标读者：Aetherlink 开发者。
> 结论先行：**照抄工作区的「双轨」骨架——同一个领域核心，UI（人用）和聊天（模型用）两条访问轨道共用一套 service；智能体轨道从第一天就把字段/作用域留好，但暂不实现。检索走 agentic 工具调用（`kb_*`），不做自动 RAG 塞 prompt。**

---

## 0. 背景：先对齐工作区的「双轨」范式

知识库不是凭空设计，而是**照抄本仓库工作区已经跑通的骨架**。先把工作区的双轨说清楚，知识库就是同一套模式换个领域：

- **轨道 A（人用 / UI）**：`WorkspaceStore`（`lib/features/workspace/application/workspace_store.dart`）+ `currentWorkspaceProvider`（`workspace_view_providers.dart`）+ 文件树 / 查看器页面（`presentation/mobile/`）。
- **轨道 B（模型用 / 聊天）**：内置 MCP 服务器 `@aether/file-editor`（在 `lib/shared/config/builtin_mcp_servers.dart` 注册，id `builtin-file-editor`），执行走 `runFileEditorTool(ref, ...)`（`lib/shared/mcp_tools/file_editor/file_editor_tools.dart`），在 `chat_controller` 的 `_mcpSetup()` 里按开关注入、在 `_runTool()` 里路由。这条轨道和 UI **共用同一批 Riverpod provider / backend**。
- **轨道 C（智能体，未设计）**：file-editor 工具注释里已埋了 “agent tools” 字样，说明工具核心是复用的（参见《工作区与智能体模式-设计构想》阶段 B）。

**知识库要做的**：同一个领域（知识库），三条访问轨道共用一套核心 service。现在只点亮轨道 A + B，轨道 C 留字段。

---

## 1. 设计目标与约束

1. **对齐工作区三段式**：`Store（元数据） + Provider + 内置 MCP 工具`。UI 和聊天共用一套 service，不分叉。
2. **agentic，不做自动 RAG**：把知识库变成模型**主动调用**的工具（`kb_list / kb_search / kb_read / kb_manage`），而不是检索完自动塞进 prompt。这与仓库现有 `_maybeInjectMemorySearch`（`chat_controller.dart`）把「记忆搜索」作为工具注入的做法一致。
3. **移动端现实**：单 Dart isolate、embedding 是付费 + 联网的、要省电。→ **复用现成的 `EmbeddingService`（`lib/features/memory/data/embedding_service.dart`）+ `SqliteVecService`（`.../local/sqlite_vec_service.dart`）**；嵌入走惰性 + 增量，永远有关键词兜底。
4. **智能体预留**：数据模型与作用域从第一天为智能体留绑定位（`scope.agentIds`），本期不实现。
5. **复用统一引用**：检索结果直接产出仓库已有的 `KnowledgeReferenceItem`（`lib/features/chat/domain/entities/knowledge_reference_item.dart`）→ `KnowledgeReferenceBlock` 引用卡片零成本渲染（`presentation/widgets/blocks/data_blocks.dart`）。

> 参照对象：Cherry Studio v2 的知识库（主库 + 每库独立 `index.sqlite`、`searchKnowledge(query, baseIds, allowedIds)`、`assistant.knowledgeBaseIds` 作用域、RRF 融合、conceptId 寻址、按内容哈希去重不重复扣费）。下文凡「照抄 CS」即指此。

---

## 2. 双轨作用域模型（核心抽象）

让知识库「像工作区一样双轨、且未来能给智能体」的关键：定义一个**作用域绑定**，而不是把「给谁用」硬编码。

```dart
class KnowledgeScope {
  final bool chatEnabled;        // 轨道 B：是否对普通聊天暴露 kb_* 工具
  final List<String> agentIds;   // 轨道 C：绑定到哪些智能体（本期恒为空，预留）
}
```

- 普通聊天读 `chatEnabled == true` 的库集合（类比工作区「最近打开」列表能被 file-editor 看到）。
- 智能体将来读 `agentIds.contains(currentAgentId)` 的库——本期不实现，字段先落库。
- 检索核心签名照抄 CS：`searchKnowledge(query, baseIds, {allowedIds})`。`allowedIds` 空 = 全部（聊天轨道）；非空 = 智能体作用域。**同一份 `searchKnowledge`，两条轨道只是传不同的 `allowedIds`。**

---

## 3. 数据模型（CS 双存储 → 单 Drift 库落地）

CS 是「主库 + 每库独立 `index.sqlite`」。移动端不适合几十个 sqlite 文件（句柄 / 管理开销），改成**一个 Drift 库内的分层表 + 持久化 sqlite-vec 虚拟表**。

### 3.1 权威元数据（Drift 表，不可从别处重建）

- **`knowledge_base`**：`id / name / embeddingModelKey(providerId:modelId) / dimensions / chunkSize / chunkOverlap / searchMode(vector|keyword|hybrid) / threshold / topK / status / scope(JSON: {chatEnabled, agentIds}) / createdAt`。
- **`knowledge_item`**：`id / baseId / type(file|url|note|workspace) / source / status(idle→reading→chunking→embedding→completed|failed) / conceptId(稳定相对路径寻址) / error`。

状态机与 `conceptId` 寻址完全沿用 CS。

### 3.2 派生索引（可从元数据 + 原文重建）

- **`kb_chunk`**：`chunkId / baseId / itemId / unitIndex / charStart / charEnd / text / contentHash / embeddingHash`。保持 CS 不变式：`fullText.substring(charStart, charEnd) == text`。
- **`kb_embedding`**：`embeddingHash / vectorBlob(float32)`，**按哈希去重**（CS 的“不变内容不重复扣费”）。
- **sqlite-vec `vec0` 持久虚拟表**：把现在 `SqliteVecService` 的**临时内存索引**（`topKNearest` 里每次 `openInMemory()` 建临时 `vec0`）升级为持久表：
  ```sql
  CREATE VIRTUAL TABLE kb_vec USING vec0(chunkId TEXT PRIMARY KEY, embedding FLOAT[dim]);
  ```
  不可用时自动回退 Dart cosine（`SqliteVecService` 已有 `probe()` + 全程降级为 `null`/`false` 的纪律，永不抛）。

> 派生表随时可从 `knowledge_item` + 原文重建，这点与 CS 一致；崩溃恢复时重建即可。

## 4. 摄取管线（移动端友好版）

CS 用 JobManager 队列 + Electron 主进程让步。本仓库没有多进程，改为：切块丢 `compute()`/isolate（纯函数），主 isolate 串行嵌入，配一个**轻量 per-base 串行队列**（async-mutex，类比 SSH 池的串行化模式）。

```
addItem → status 流转:
  reading   : file 读盘 / url 抓取存快照 / note 存内容 / workspace 遍历后端文件
  chunking  : 结构感知切块（移植 CS splitter，纯函数，可丢 isolate）
  embedding : 惰性——只 embed kb_embedding 里缺失的 embeddingHash；失败不 abort，标 failed 可重试
  completed : 原子写入 kb_chunk / kb_embedding / kb_vec
```

**独创点：`type: workspace` 源。** 知识库可直接把一个工作区目录当作源——复用 `WorkspaceBackend.listDir/readFile` 遍历文件建索引。这样两个双轨功能打通：**工作区负责读写文件，知识库负责语义检索同一批文件。**

嵌入策略照抄 memory 的惰性 backfill（见 `lib/app/di/memory_access.dart` 的 `_semanticTopK`）：无嵌入模型 / 嵌入失败 → 自动降级关键词检索，**绝不因检索失败打断对话**。

---

## 5. 检索（hybrid + 复用现有 sqlite-vec）

移植 CS 三模式，但用现有基建：

- **vector**：`SqliteVecService.topKNearest`（升级为持久 `kb_vec`）→ 不可用则 Dart cosine（`rankBySimilarity`）。
- **keyword**：Drift 上对 `kb_chunk.text` 做 `LIKE`（移动端先不用 FTS5，够用；中文短词天然走子串，正好避开 CS 那个 trigram 分词坑）。
- **hybrid**：两路 RRF 融合（CS 的 `fuseWithRrf` 是纯函数，直接搬）。

**流程**：超取 → 可见性过滤（回 `knowledge_item` 校验 `completed` + 属本库 + 在作用域内）→ 裁 `topK` → 阈值过滤 → 产出 `KnowledgeReferenceItem`（字段现成：`index / content / similarity / documentId / knowledgeBaseId / knowledgeBaseName / sourceUrl`）。

## 6. 与聊天集成（内置 MCP 服务器 `@aether/knowledge`）

完全套用 file-editor 的落地方式：

- **catalog**：`builtin_mcp_servers.dart` 加一条 `McpServer(id: 'builtin-knowledge', name: '@aether/knowledge', type: inMemory, category: builtin, ...)`；`builtin_tool_catalog.dart` 加 4 个工具定义 `kb_list / kb_search / kb_read / kb_manage`。
- **执行**：新建 `runKnowledgeTool(ref, toolName, args)`，因需 `Ref` 读 provider，加入 `kRefDependentBuiltins`（`builtin_tool_catalog.dart`）；在 `chat_controller._runTool()` 加一个 `_KnowledgeToolRoute` 分支（对齐 file-editor 的 `runFileEditorTool` 路由）。
- **风险门控**：`kb_manage` 的 `add/delete/refresh` 标 high-risk，走现成 HITL 确认（对齐 `fileEditorRiskLevel`，`lib/shared/mcp_tools/file_editor/file_editor_tools.dart`）。
- **作用域注入**：像 `_maybeInjectMemorySearch` 那样——即使 MCP 总开关关着，只要有 `chatEnabled` 的库就注入 `kb_search`。这就是「提供给普通聊天」的开关。

做完这步，知识库的**聊天轨道**即通，和工作区现状对等。

## 7. 智能体接入预留（本期不写，字段先留）

- `knowledge_base.scope.agentIds` 本期恒空。
- `searchKnowledge` 的 `allowedIds` 聊天轨道传空（= 全部 `chatEnabled` 库）。
- 等智能体设计好：智能体持有 `knowledgeBaseIds`，调用时把它作为 `allowedIds` 传进**同一个** `searchKnowledge`。核心零改动。

---

## 8. 独创点总结（相对 Cherry Studio 的差异）

1. **双轨作用域抽象** `KnowledgeScope{chatEnabled, agentIds}`：一套核心两条轨道，智能体即插即用。
2. **`workspace` 作为知识源**：打通两个双轨功能，工作区文件可被语义检索。
3. **单 Drift 库 + 持久 sqlite-vec**：把 memory 的临时 `vec0` 索引升级为持久层，避免 CS 多文件方案在移动端的句柄 / 管理开销。
4. **移动端惰性嵌入 + 关键词永不失败兜底**：复用 memory 已验证的降级路径。
5. **零成本引用渲染**：检索直接产出 `KnowledgeReferenceItem`，`KnowledgeReferenceBlock` 已能画。

## 9. 落地路线

| 阶段 | 内容 | 产出 |
|---|---|---|
| **P0** | Drift 表 + `KnowledgeBaseStore` / `KnowledgeService` 骨架 + 摄取（note/file）+ 关键词检索 | 能建库、加文档、纯关键词搜到 |
| **P1** | 接入 `EmbeddingService` + 持久 sqlite-vec + hybrid + 产出引用 | 语义检索可用 |
| **P2** | 内置 MCP `@aether/knowledge` 4 工具 + 聊天注入 + HITL | 聊天轨道打通 |
| **P3** | `workspace` 源、目录展开、重索引、崩溃恢复 | 功能完整 |
| **P4** | 智能体作用域绑定（等智能体设计好） | 智能体轨道 |

## 10. 待决策 / 风险点

- **嵌入模型选择与维度**：`embeddingModelKey` + `dimensions` 落库后不可随意改；换模型 = 重建全部派生索引（CS 同）。UI 需明确提示。
- **sqlite-vec 真机可用性**：`SqliteVecService.probe()` 在 `flutter test` 里恒 `available: false`（无插件注册），只能真机确认——持久 `kb_vec` 表要在探测可用后才启用，不可用时回退 Dart cosine，且**迁移不可依赖 vec0 存在**。
- **切块 isolate 的依赖边界**：切块须是纯函数（不碰 Drift / provider），才能安全丢 `compute()`。
- **中文检索**：`LIKE` 子串对中文够用，但长文档召回质量仍依赖 embedding；关键词仅兜底，勿当主力。
- **HITL 粒度**：`kb_manage` 的删除 / 重索引是破坏性/花钱操作，必须确认；`kb_search/kb_read` 只读免确认。

---

## 11. 一句话总结

> 知识库 = 工作区那套「Store + Provider + 内置 MCP 工具」三段式换到「知识」领域；
> 用 `KnowledgeScope{chatEnabled, agentIds}` 一套核心撑起「人用 / 聊天 / 智能体」三轨；
> 复用 memory 的 `EmbeddingService` + `SqliteVecService`，惰性嵌入、关键词兜底、持久化 sqlite-vec；
> 先点亮聊天轨道（`@aether/knowledge` 的 `kb_*` 工具），智能体轨道留字段等设计。
