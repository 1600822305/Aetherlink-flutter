# 知识库（Knowledge Base）功能设计构想

> 状态：设计草案，尚未落地（当前仅有输入框「知识库」按钮的 UI 占位，点击即「敬请期待」）。
> 目标读者：Aetherlink 开发者
> 参考：Cherry Studio 桌面版知识库架构（双存储 + agentic 工具），并对齐本项目《工作区与智能体模式-设计构想》的双轨范式。
> 结论先行：**知识库和工作区是同构的两个领域——都要"人用的 UI 轨道"与"模型用的工具轨道"共享同一套 Store/Service，且都为尚未设计的智能体预留绑定位。核心检索/摄取逻辑一次实现，三条轨道（UI、聊天、未来智能体）复用。**

---

## 0. 背景与目标

Aetherlink 现状：

1. **工作区**（`features/workspace/`）已落地 P0——`WorkspaceBackend` 抽象 + 本地 SAF 实现，UI 轨道（文件树/查看器）与聊天轨道（`@aether/file-editor` 内置 MCP 工具）都已打通，双轨共用同一个 `WorkspaceStore`。
2. **聊天记忆**（`features/memory/`）已落地——`EmbeddingService`（OpenAI 兼容 `/embeddings`）+ `SqliteVecService`（sqlite-vec 原生 KNN，off 则退化 Dart 余弦）+ Drift 存储，全部验证可用。
3. **知识库**：只有输入框按钮 + 聊天消息里的引用渲染组件（`KnowledgeReferenceBlock`/`KnowledgeReferenceItem`，用于显示引用卡片），**没有任何建库、摄取、检索的实现**。

目标：设计一套知识库功能——

- 复用①的双轨范式（UI 轨道 + 内置 MCP 工具轨道）；
- 复用②已验证的 embedding / 向量检索基建；
- 现在只点亮"聊天轨道"，但数据模型从第一天起就给"智能体轨道"（尚未设计）留好绑定位，不推倒重来。

---

## 1. 核心原则

### 1.1 Agentic，不做自动 RAG 注入

不在发送前静默检索、把结果塞进 system prompt。改为给模型 `kb_list / kb_search / kb_read / kb_manage` 四个工具，模型自己决定要不要查、查哪个库、要不要深读原文——检索失败或没结果时模型能感知并追问/换策略，而不是拿到一段不明来源的"背景资料"。

### 1.2 双存储：权威元数据 + 可重建派生索引

- **权威**：知识库配置、文档条目、生命周期状态——存 Drift 表，是唯一真相源。
- **派生**：切块、向量、Embedding——可以从权威数据 + 原文**随时重建**的缓存层，schema 升级或损坏时直接重建而非在线迁移。

### 1.3 双轨作用域，而不是硬编码"给谁用"

参照工作区"UI 打开的工作区 = 聊天 `@aether/file-editor` 能看到的工作区"，知识库定义一个作用域字段，同一份检索核心供多轨复用（见第 3 节），而不是给聊天和智能体各写一套检索逻辑。

### 1.4 移动端务实

- 没有 Electron 主进程/任务队列，摄取管线用 Dart 层的**串行队列 + isolate 切块**，而非常驻后台服务。
- Embedding 是**联网 + 付费**的：按内容哈希去重、惰性补算、失败自动降级关键词检索——绝不能因为一次网络抖动让检索整体失败。
- 向量检索优先复用 `SqliteVecService`（已验证的 sqlite-vec 持久化 KNN），不可用时退化 Dart 余弦，这条降级路径记忆功能已经跑通。

## 2. 总体架构

```
┌──────────────────────────────────────────────────────────┐
│  轨道层   UI 页面(建库/管理)   @aether/knowledge 工具(聊天)  │
│           │ 共享 Store/Service                             │
├──────────────────────────────────────────────────────────┤
│  作用域层 KnowledgeScope { chatEnabled, agentIds }          │
├──────────────────────────────────────────────────────────┤
│  核心层   KnowledgeService                                  │
│    摄取(读取→切块→嵌入→写入) · 检索(vector/keyword/hybrid)  │
│    · 深读(kb_read) · 管理(kb_manage)                        │
├──────────────────────────────────────────────────────────┤
│  存储层   权威: knowledge_base / knowledge_item (Drift)     │
│           派生: kb_chunk / kb_embedding / kb_vec(sqlite-vec)│
├──────────────────────────────────────────────────────────┤
│  基建层   EmbeddingService · SqliteVecService · WorkspaceBackend(可选源) │
└──────────────────────────────────────────────────────────┘
```

对齐工作区的分层方式：轨道层可以有多个入口，但都收敛到同一个核心层；核心层不知道调用方是 UI 还是模型。

## 3. 双轨作用域模型（对齐工作区双轨）

```dart
class KnowledgeScope {
  const KnowledgeScope({
    this.chatEnabled = true,
    this.agentIds = const <String>[],
  });

  /// 轨道 B：是否对普通聊天暴露（对应工作区"最近打开"即对 file-editor 可见）。
  final bool chatEnabled;

  /// 轨道 C：绑定的智能体 id 列表。现在恒为空——智能体尚未设计，
  /// 字段先落库，避免智能体落地时改 schema。
  final List<String> agentIds;
}
```

- **聊天轨道**调用 `KnowledgeService.search(query, baseIds, allowedIds: null)`：`allowedIds == null` 表示"聊天可见"，内部过滤 `scope.chatEnabled == true` 的库。
- **智能体轨道**（预留）：等智能体设计出 `knowledgeBaseIds` 绑定后，直接把它当 `allowedIds` 传进**同一个** `search` 方法——核心检索代码零改动，这一点严格照抄 Cherry Studio `assistant.knowledgeBaseIds` 的收敛方式。

## 4. 数据模型

### 4.1 权威表（Drift，唯一真相源）

**`knowledge_base`**

| 列 | 说明 |
|---|---|
| `id` | 主键 |
| `name` | 显示名 |
| `embeddingModelKey` | `providerId:modelId`，复用记忆功能的模型选择方式 |
| `dimensions` | 嵌入维度，首次成功嵌入后写入并锁定 |
| `chunkSize` / `chunkOverlap` | 切块参数 |
| `searchMode` | `vector \| keyword \| hybrid` |
| `threshold` / `topK` | 检索参数 |
| `scope` | `KnowledgeScope` 的 JSON 编码 |
| `status` | `idle \| indexing \| completed \| failed` |
| `createdAt` |  |

**`knowledge_item`**

| 列 | 说明 |
|---|---|
| `id` | 主键 |
| `baseId` | 所属库 |
| `type` | `file \| url \| note \| workspace`（见 §8） |
| `source` | 原始来源（路径 / URL / workspace root） |
| `conceptId` | 稳定寻址键（相对路径），供 `kb_read` 精确定位 |
| `status` | `idle → reading → chunking → embedding → completed \| failed` |
| `error` | 失败原因 |

### 4.2 派生表（可重建，Drift + sqlite-vec）

**`kb_chunk`**：`chunkId / baseId / itemId / unitIndex / charStart / charEnd / text / contentHash / embeddingHash`。

保持不变式 `原文.slice(charStart, charEnd) == text`——正文只存一份（挂在 `knowledge_item` 对应的快照文件或 note 内容上），chunk 是切片派生，避免重复存储。

**`kb_embedding`**：`embeddingHash(PK) / vectorBlob`。按内容哈希去重——同一段文本不重复付费嵌入，重新索引未变内容不再扣费。

**`kb_vec`（sqlite-vec 虚拟表）**：`CREATE VIRTUAL TABLE kb_vec USING vec0(chunkId TEXT PRIMARY KEY, embedding FLOAT[dim])`。把 `SqliteVecService` 现有的"每次查询建临时内存索引"升级为持久表；`SqliteVecService.available == false` 时全链路退化到 Dart 余弦（`kb_embedding` 全表扫描），这条路径与记忆功能共享同一个 probe 结果。

## 5. 摄取管线

```
addItem(type, source)
  → reading    : file 读盘 / url 抓取转 Markdown 存快照 / note 存内容快照 /
                 workspace 通过 WorkspaceBackend.listDir+readFile 遍历
  → chunking   : 结构感知切块（移植 Cherry Studio 的 markdown 断点评分算法），
                 纯函数，丢进 compute() isolate 跑，避免卡主 isolate
  → embedding  : 只对 kb_embedding 缺失的 embeddingHash 调 EmbeddingService；
                 单条失败不 abort 整个 item，标记可重试
  → completed  : 原子写 kb_chunk + kb_embedding + kb_vec；状态机翻转
```

- 摄取是**每库一条串行队列**（简单的 Dart `Future` 链即可，参考 `SshBackendPool` 的 per-connection 隔离思路），避免同库并发写 sqlite-vec 表。
- 嵌入模型未配置 / 请求失败 → `knowledge_base.searchMode` 自动降级为 `keyword`，UI 提示，不阻断摄取本身（关键词检索仍然建得起来）。

---

## 6. 检索

三种模式，`hybrid` 用 RRF（Reciprocal Rank Fusion）融合排名而非直接比较分数量纲：

- **vector**：`SqliteVecService.topKNearest` → 不可用时 `rankBySimilarity`（Dart 余弦，记忆功能已有实现，直接复用）。
- **keyword**：Drift 上对 `kb_chunk.text` 做 `LIKE '%token%'`（移动端数据量小，暂不引入 FTS5；天然覆盖中文短词，避开桌面版 trigram 分词对 1-2 字词失效的问题）。
- **hybrid**：两路各取候选，按排名做 RRF 融合。

统一流程：**超取候选 → 按 `KnowledgeScope` 与 `knowledge_item.status == completed` 过滤可见性 → 裁剪 topK → 阈值过滤 → 产出 `KnowledgeReferenceItem`**（复用聊天领域已有的引用类型，`KnowledgeReferenceBlockView` 零改动即可渲染结果）。

## 7. 与聊天集成（轨道 B：内置 MCP 服务器）

完全复刻 `@aether/file-editor` 的落地方式：

1. **目录**：`builtin_mcp_servers.dart` 新增 `@aether/knowledge`；`builtin_tool_catalog.dart` 补 4 个工具定义：
   - `kb_list`：列出可见知识库，或给定 baseId 输出目录树。
   - `kb_search`：语义/关键词检索，返回 `KnowledgeReferenceItem[]`。
   - `kb_read`：按 `conceptId` 读整篇 / 按正则 grep 精确定位。
   - `kb_manage`：增/删/重索引（写操作）。
2. **执行**：新建 `runKnowledgeTool(ref, toolName, args)`，加入 `kRefDependentBuiltins`（需要 `Ref` 读 `KnowledgeService` provider），在 `chat_controller._runTool()` 加 `_KnowledgeToolRoute` 分支，模式与 `_FileEditorToolRoute` 完全一致。
3. **风险门控**：`kb_manage` 的 add/delete/refresh 标记为需要 HITL 确认，复用 `fileEditorRiskLevel` 同款风险分级机制。
4. **注入开关**：`chatEnabled` 的库存在时即可注入 `kb_search`/`kb_list`，可比照 `_maybeInjectMemorySearch` 做成独立于 MCP 总开关的注入位，也可以走标准 MCP 服务器目录（二者都支持，先按标准内置服务器落地更简单）。

聊天轨道打通后，知识库与工作区在"暴露给聊天"这件事上完全对等。

---

## 8. 独创点：workspace 作为知识源

`knowledge_item.type == workspace` 时，`source` 存一个 `Workspace.id`，摄取阶段通过 `WorkspaceBackend.listDir` / `readFile` 遍历该工作区目录建索引。

这把项目里两个"双轨"领域打通：工作区负责"读写文件"，知识库负责"对同一批文件做语义检索"——用户打开的项目目录，不用手动逐个添加文件，一键就能被聊天语义搜索到。技术上零风险：只是多了一种 `knowledge_item` 来源，摄取/检索/工具三层完全不用感知来源差异。

---

## 9. 智能体轨道预留（现在不实现）

- `knowledge_base.scope.agentIds` 现在恒为空数组，仅落库。
- `KnowledgeService.search` 的 `allowedIds` 参数现在聊天轨道恒传 `null`。
- 等《工作区与智能体模式-设计构想》里的智能体领域落地、`AgentTask`/`Assistant` 一类实体有了 `knowledgeBaseIds` 绑定字段后，直接传入 `allowedIds`——核心检索/摄取代码不需要任何改动，这是本设计从一开始就要保证的收敛点。

---

## 10. 落地顺序（建议）

- ⬜ **P0 骨架**：`knowledge_base` / `knowledge_item` Drift 表 + `KnowledgeBaseStore`（仿 `WorkspaceStore`）+ note/file 摄取 + 纯关键词检索，UI 先只做"建库/加笔记/搜索"最小闭环。
- ⬜ **P1 语义检索**：接入 `EmbeddingService` + 持久化 `kb_vec` + hybrid 检索 + 产出 `KnowledgeReferenceItem`。
- ⬜ **P2 聊天轨道**：内置 MCP `@aether/knowledge` 四工具 + `chat_controller` 接入 + HITL 确认——**知识库对齐工作区当前双轨水平**。
- ⬜ **P3 完整摄取源**：url 抓取、workspace 源（§8）、重索引、失败恢复。
- ⬜ **P4 智能体轨道**：等智能体领域设计完成后，接入 `allowedIds` 绑定（见 §9）。

---

## 11. 需要尽早决策的风险点

- **移动端 sqlite-vec 可用性**：`SqliteVecService.probe()` 在部分设备/架构上可能失败，必须保证 Dart 余弦全链路无缝兜底，且性能在几千 chunk 量级下可接受。
- **切块算法的移植成本**：Cherry Studio 的结构感知切块（markdown 断点评分 + 代码围栏保护）价值高但实现量不小，P0 可先用简单定长切块，P1 再升级，不阻塞骨架落地。
- **Embedding 成本可见性**：需在 UI 上明确提示"重新索引未变内容不重复扣费"，管理好用户预期（同记忆功能的哈希去重口径）。
- **workspace 源的一致性**：工作区文件可能被外部改动（SAF 只能感知"通过本 backend 的改动"），knowledge_item 需要有"过期/需重索引"提示，不能默默用旧索引回答。

---

## 12. 一句话总结

> 知识库不是"聊天的附加功能"，是和工作区同构的独立领域：
> 用一个 `KnowledgeScope` 统一"聊天可见 / 智能体可见"，核心摄取与检索逻辑只写一份；
> 复用记忆功能已验证的 Embedding + sqlite-vec 基建，复用工作区已验证的"Store + 内置 MCP 工具"双轨落地方式；
> 先打通聊天轨道，智能体轨道等其领域设计完成后即插即用。
