# 知识库（Knowledge Base）功能设计构想

> 状态：设计草案 v2（已过一轮自我批判修订），尚未落地（当前仅有输入框「知识库」按钮的 UI 占位，点击即「敬请期待」）。
> 目标读者：Aetherlink 开发者
> 参考：Cherry Studio 桌面版知识库架构（双存储 + agentic 工具），并对齐本项目《工作区与智能体模式-设计构想》的双轨范式。
> 结论先行：**知识库和工作区是同构的两个领域——都要"人用的 UI 轨道"与"模型用的工具轨道"共享同一套 Store/Service，且都为尚未设计的智能体预留绑定位。核心检索/摄取逻辑一次实现，三条轨道（UI、聊天、未来智能体）复用。**

## 0. 背景与目标

Aetherlink 现状：

1. **工作区**（`features/workspace/`）已落地 P0——`WorkspaceBackend` 抽象 + 本地 SAF 实现，UI 轨道（文件树/查看器）与聊天轨道（`@aether/file-editor` 内置 MCP 工具）都已打通，双轨共用同一个 `WorkspaceStore`。
2. **聊天记忆**（`features/memory/`）已落地——`EmbeddingService`（OpenAI 兼容 `/embeddings`）+ `SqliteVecService`（sqlite-vec 原生 KNN，off 则退化 Dart 余弦）+ Drift 存储，全部验证可用。
3. **知识库**：只有输入框按钮 + 聊天消息里的引用渲染组件（`KnowledgeReferenceBlock`/`KnowledgeReferenceItem`，用于显示引用卡片），**没有任何建库、摄取、检索的实现**。

目标：设计一套知识库功能——

- 复用①的双轨范式（UI 轨道 + 内置 MCP 工具轨道）；
- 复用②已验证的 embedding / 向量检索基建；
- 现在只点亮"聊天轨道"，但数据模型从第一天起就给"智能体轨道"（尚未设计）留好绑定位，不推倒重来。

## 1. 核心原则

### 1.1 Agentic，不做自动 RAG 注入

不在发送前静默检索、把结果塞进 system prompt。改为给模型 `kb_list / kb_search / kb_read / kb_manage` 四个工具，模型自己决定要不要查、查哪个库、要不要深读原文——检索失败或没结果时模型能感知并追问/换策略，而不是拿到一段不明来源的"背景资料"。

### 1.2 双存储：权威元数据 + 可重建派生索引

- **权威**：知识库配置、文档条目、生命周期状态——存 Drift 表，是唯一真相源。
- **派生**：切块、向量、Embedding——可以从权威数据 + 原文**随时重建**的缓存层，schema 升级或损坏时直接重建而非在线迁移。

### 1.3 双轨作用域，而不是硬编码"给谁用"

参照工作区"UI 打开的工作区 = 聊天 `@aether/file-editor` 能看到的工作区"，知识库定义一个作用域字段，同一份检索核心供多轨复用（见第 3 节），而不是给聊天和智能体各写一套检索逻辑。

### 1.4 移动端务实

- 没有 Electron 主进程/任务队列，摄取管线用 Dart 层的**串行队列 + isolate 切块**，而非常驻后台服务。
- Embedding 是**联网 + 付费**的：按内容哈希去重、惰性补算、失败自动降级关键词检索——绝不能因为一次网络抖动让检索整体失败。
- 向量检索优先复用 `SqliteVecService`（已验证的 sqlite-vec 持久化 KNN），不可用时退化 Dart 余弦，这条降级路径记忆功能已经跑通。

## 2. 总体架构

```
┌──────────────────────────────────────────────────────────┐
│  轨道层   UI 页面(建库/管理)   @aether/knowledge 工具(聊天)  │
│           │ 共享 Store/Service                             │
├──────────────────────────────────────────────────────────┤
│  作用域层 KnowledgeScope { chatEnabled, agentIds }          │
├──────────────────────────────────────────────────────────┤
│  核心层   KnowledgeService                                  │
│    摄取(读取→切块→嵌入→写入) · 检索(vector/keyword/hybrid)  │
│    · 深读(kb_read) · 管理(kb_manage)                        │
├──────────────────────────────────────────────────────────┤
│  存储层   权威: knowledge_base / knowledge_item (Drift)     │
│           派生: kb_chunk / kb_embedding / kb_vec(sqlite-vec)│
├──────────────────────────────────────────────────────────┤
│  基建层   EmbeddingService · SqliteVecService · WorkspaceBackend(可选源) │
└──────────────────────────────────────────────────────────┘
```

对齐工作区的分层方式：轨道层可以有多个入口，但都收敛到同一个核心层；核心层不知道调用方是 UI 还是模型。

## 3. 双轨作用域模型（对齐工作区双轨）

```dart
class KnowledgeScope {
  const KnowledgeScope({
    this.chatEnabled = true,
    this.agentIds = const <String>[],
  });

  /// 轨道 B：是否对普通聊天暴露（对应工作区"最近打开"即对 file-editor 可见）。
  final bool chatEnabled;

  /// 轨道 C：绑定的智能体 id 列表。现在恒为空——智能体尚未设计，
  /// 字段先落库，避免智能体落地时改 schema。
  final List<String> agentIds;
}
```

- **聊天轨道**调用 `KnowledgeService.search(query, baseIds, allowedIds: null)`：`allowedIds == null` 表示"聊天可见"，内部过滤 `scope.chatEnabled == true` 的库。
- **智能体轨道**（预留）：等智能体设计出 `knowledgeBaseIds` 绑定后，直接把它当 `allowedIds` 传进**同一个** `search` 方法——核心检索代码零改动，这一点严格照抄 Cherry Studio `assistant.knowledgeBaseIds` 的收敛方式。

## 4. 数据模型

### 4.1 权威表（Drift，唯一真相源）

**`knowledge_base`**

| 列 | 说明 |
|---|---|
| `id` | 主键 |
| `name` | 显示名 |
| `embeddingModelKey` | `providerId:modelId`，复用记忆功能的模型选择方式 |
| `dimensions` | 嵌入维度，首次成功嵌入后写入并锁定 |
| `chunkSize` / `chunkOverlap` | 切块参数 |
| `searchMode` | `vector \| keyword \| hybrid` |
| `threshold` / `topK` | 检索参数 |
| `scope` | `KnowledgeScope` 的 JSON 编码 |
| `status` | `idle \| indexing \| completed \| failed` |
| `createdAt` |  |

**`knowledge_item`**

| 列 | 说明 |
|---|---|
| `id` | 主键 |
| `baseId` | 所属库 |
| `type` | `file \| url \| note \| workspace`（见 §8） |
| `source` | 原始来源（路径 / URL / workspace root） |
| `conceptId` | 稳定寻址键（相对路径），供 `kb_read` 精确定位 |
| `status` | `idle → reading → chunking → embedding → completed \| failed` |
| `error` | 失败原因 |

### 4.2 正文存储：`knowledge_content`

新增一张权威表，明确"原文存哪"——不再含糊地说"挂在快照文件或 note 内容上"：

| 列 | 说明 |
|---|---|
| `itemId(PK/FK)` | 一对一挂在 `knowledge_item` 上 |
| `text` | 完整正文。note / url 快照 / file 转换后的文本统一走这里 |
| `contentHash` | 正文的 sha256，供 §4.3 去重判断"内容是否变化" |

- 阈值以下（如 < 1MB）直接存 Drift `text` 列；超过阈值的大文件正文落盘到 `app_documents/kb/<baseId>/<itemId>.md`，该列改存文件相对路径 + 一个 `isFile` 标记。两种情况对上层（切块、`kb_read`）透明，读取时统一走一个 `readItemContent(itemId)` 函数屏蔽差异。
- 源文件本体（用户原始 PDF/图片等，转换前）另放 `app_documents/kb/<baseId>/raw/<itemId>.<ext>`，与转换后的正文快照分离，便于以后换更好的转换器重新摄取。

### 4.3 派生表（可重建，Drift + 每库独立 sqlite-vec 表）

**`kb_chunk`**：`chunkId / baseId / itemId / unitIndex / charStart / charEnd / text / contentHash / embeddingKey`。

保持不变式 `knowledge_content.text.slice(charStart, charEnd) == text`——chunk 是切片派生，不重复存正文。

**`kb_embedding`**：`embeddingKey(PK) / vectorBlob`，其中 **`embeddingKey = sha256(embeddingModelKey + '|' + contentHash)`**（复合键，而非单纯内容哈希）。

> **关键修正**：同一段文本在两个用不同 embedding 模型的知识库里，向量维度不同——纯内容哈希去重会导致跨模型的向量被错误复用/覆盖。去重范围必须限定在"同一个 embedding 模型内"，所以键里必须带 `embeddingModelKey`。

**`kb_vec_<dimensions>`（sqlite-vec 虚拟表，按维度分表，而非全局一张表）**：

```sql
CREATE VIRTUAL TABLE kb_vec_1536 USING vec0(embedding_key TEXT PRIMARY KEY, embedding FLOAT[1536]);
CREATE VIRTUAL TABLE kb_vec_768  USING vec0(embedding_key TEXT PRIMARY KEY, embedding FLOAT[768]);
```

> **关键修正**：`vec0` 虚拟表列宽在建表时固定，不能一张表塞不同维度的向量。表名按 `dimensions` 动态派生（`kb_vec_$dimensions`），首次出现该维度时建表，之后复用；一个知识库固定用其 `dimensions` 对应的那张表。`SqliteVecService.available == false` 时全链路退化到 Dart 余弦（对 `kb_embedding` 按 `embeddingKey` 前缀过滤出同模型向量再算），这条降级路径与记忆功能共享同一个 probe 结果。

## 5. 摄取管线

```
addItem(type, source)
  → reading    : file 读盘 / url 抓取转 Markdown 存快照 / note 存内容快照 /
                 workspace 通过 WorkspaceBackend.listDir+readFile 遍历
  → chunking   : 结构感知切块（移植 Cherry Studio 的 markdown 断点评分算法），
                 纯函数，丢进 compute() isolate 跑，避免卡主 isolate
  → embedding  : 只对 kb_embedding 缺失的 embeddingKey 调 EmbeddingService；
                 单条失败不 abort 整个 item，标记可重试
  → completed  : 原子写 kb_chunk + kb_embedding + kb_vec_<dim>；状态机翻转（见 §5.1）
```

- 摄取是**每库一条串行队列**（简单的 Dart `Future` 链即可，参考 `SshBackendPool` 的 per-connection 隔离思路），避免同库并发写 sqlite-vec 表。
- 嵌入模型未配置 / 请求失败 → `knowledge_base.searchMode` 自动降级为 `keyword`，UI 提示，不阻断摄取本身（关键词检索仍然建得起来）。
- **富文本转换范围先明确收窄**：P0/P1 仅支持 `txt / md / note / url`（url 走现成的网页转 Markdown）。PDF / DOCX 等需要额外转换器，移动端没有 Cherry Studio 那种主进程可跑的转换库，**明确列为 P3e 待做**，不在 P0 承诺范围内，避免被误解为"隐式已支持"。P3e 方案见 §5.2。

### 5.1 重索引的原子性

Cherry Studio 用单个 sqlite 事务原子替换某文档的全部派生数据（`rebuildMaterial`）。移动端有一个额外的坑：**sqlite-vec 走的是原生 sqlite3 连接，不在 Drift 的事务管理范围内**——Drift 提交成功但 vec0 写入失败会产生不一致（chunk 有了，向量没有）。

约定的重建顺序（`itemId` 粒度，可重试、非严格原子但保证"宁可缺失也不脏"）：

1. 删该 `itemId` 在 `kb_vec_<dim>` 里的旧行（按 `embeddingKey` 前缀）。
2. Drift 事务内：删旧 `kb_chunk`，插新 `kb_chunk` + 缺失的 `kb_embedding`。
3. 事务提交成功后，再把新增的 `kb_embedding` 写入 `kb_vec_<dim>`。
4. 第 3 步失败：`knowledge_item.status` 标记 `embedding_pending`（可关键词检索，暂无法向量检索），下次 `kb_manage refresh` 或后台重试补写。

**先删 vec 索引、再改权威三表、最后补 vec 索引**，保证任何一步中断，最坏结果是"该文档暂时搜不到向量"而不是"搜到一条指向旧内容的野向量"。

### 5.2 P3e：PDF / DOCX 解析（本地 + 云端双轨）

对齐 Cherry Studio v2 的思路：**本地解析是默认路径，云端预处理是可选增强**（CS 的 `fileProcessorId` + `knowledgeFileProcessingExts` 机制——库级配置了处理器时，文档先转 Markdown 再摄取；未配置时走本地解析器兜底）。

1. **本地解析（默认，无需任何配置）**
   - **DOCX 纯 Dart 自研**：本质 ZIP+XML，`archive`+`xml` 解析 `word/document.xml`，可在 `compute()` isolate 跑。
   - **PDF 文本层抽取**：包成熟引擎——候选 `syncfusion_flutter_pdf`（商业 License 待拍板）或 `read_pdf_text`（原生、仅移动端）。
   - 局限要在 UI 明示：扫描件 PDF（无文本层）本地解析不出内容；`.doc`/`.pptx`/`.xls` 等旧格式/复杂格式不做本地解析。
2. **云端预处理（可选，`knowledge_base` 级配置 `fileProcessorId`）**
   - 文档原文件上传到云端 OCR/转换服务（候选与 CS 相同：MinerU / Doc2X / Mistral OCR），拿回 Markdown 后**存为权威快照**再走通用 `_ingest` 摄取——与 P3b 的"URL→Markdown 快照"管线同构，`refresh` 从快照重建、不再重复调云端。
   - 对移动端反而更契合：省 CPU/内存、支持扫描件 OCR、还能覆盖本地不解析的 `.doc`/`.pptx`/`.xls`。
   - API key 走既有 provider 凭据体系；上传原文件涉及隐私，添加文档时需明确提示"将上传到 <服务名>"。
3. **降级语义**：未配置云端处理器时，`.pdf`/`.docx` 走本地解析（扫描件出不了内容时在 item 上标失败原因），其余云端专属格式直接提示"需配置文档预处理服务"。

> **实现状态（P3e 已落地）**：两轨均已实现，相对本节初稿的偏差：① PDF 引擎最终选 **pdfrx / PDFium**（BSD-3 开源，全平台）而非候选的 syncfusion（商业 EULA 禁止衍生分发）/ read_pdf_text（仅移动端），段落重排增强层自研于 `packages/pdf_to_markdown`；② 云端解析器的 API Key 存 app 设置表（按服务一份，`knowledge.file_processor.<id>.api_key`）而非复用 provider 凭据体系——三家都不是聊天 provider，强行建 provider 条目反而别扭；③ 云端专属格式（`.doc`/`.pptx`/`.xls`）的选择器接入未做，当前上传入口仍限 `txt/md/docx/pdf`，留作后续增强。

## 6. 检索

三种模式，`hybrid` 用 RRF（Reciprocal Rank Fusion）融合排名而非直接比较分数量纲：

- **vector**：`SqliteVecService.topKNearest` → 不可用时 `rankBySimilarity`（Dart 余弦，记忆功能已有实现，直接复用）。
- **keyword**：Drift 上对 `kb_chunk.text` 做 `LIKE '%token%'`（移动端数据量小，暂不引入 FTS5；天然覆盖中文短词，避开桌面版 trigram 分词对 1-2 字词失效的问题）。
- **hybrid**：两路各取候选，按排名做 RRF 融合。

统一流程：**超取候选 → 按 `KnowledgeScope` 与 `knowledge_item.status == completed` 过滤可见性 → 裁剪 topK → 阈值过滤 → 产出 `KnowledgeReferenceItem`**（复用聊天领域已有的引用类型，`KnowledgeReferenceBlockView` 零改动即可渲染结果）。

## 7. 与聊天集成（轨道 B：内置 MCP 服务器）

完全复刻 `@aether/file-editor` 的落地方式：

1. **目录**：`builtin_mcp_servers.dart` 新增 `@aether/knowledge`；`builtin_tool_catalog.dart` 补 4 个工具定义：
   - `kb_list`：无 `base_id` 时列出对聊天开放的库；给定 `base_id` 时列出该库的条目（文档）。
   - `kb_search`：语义/关键词/混合检索，可跨库融合，返回带 `documentId` 的片段（可交给 `kb_read` 取全文）。
   - `kb_read`：按 `base_id` + `document_id`（即 `kb_search` 结果里的 `documentId`）读整篇正文，超 8000 字截断。
   - `kb_manage`：`create` 建库 / `add_note` 加笔记 / `delete` 删库（写操作）。
2. **执行**：新建 `runKnowledgeTool(ref, toolName, args)`，加入 `kRefDependentBuiltins`（需要 `Ref` 读 `KnowledgeService` provider），在 `chat_controller._runTool()` 加 `_KnowledgeToolRoute` 分支，模式与 `_FileEditorToolRoute` 完全一致。
3. **风险门控**：复用 `fileEditorRiskLevel` 同款二级分级，具体到 `kb_manage` 的 action 参数：
   - `kb_list` / `kb_search` / `kb_read`：只读，**不需要**确认。
   - `kb_manage` 的 `create` / `add_note`：`medium`——写操作，走 HITL 确认（比设计初稿更保守：新增也要确认，避免模型静默建库/写库）。
   - `kb_manage` 的 `refresh`（重索引）：`medium`——P2 暂未实现，**P3a（PR #496）已落地**为 `reindexBase` 整库重建。
   - `kb_manage` 的 `delete`：`high`——真正的破坏性操作，对齐 `delete_file` 的确认强度。
4. **注入开关**：仿 `_maybeInjectMemorySearch` 落地 `_maybeInjectKnowledgeSearch`——只要存在 `chatEnabled` 的库，即便 MCP 总开关关着也注入 `kb_search`（独立于 MCP 总开关的注入位）。

聊天轨道打通后，知识库与工作区在"暴露给聊天"这件事上完全对等。

> **实现状态（P2 已落地，PR #494）**：`@aether/knowledge` 4 工具 + `_KnowledgeToolRoute` 路由/分发/HITL/确认摘要 + 系统提示 hint + `_maybeInjectKnowledgeSearch` 全部落地；`§7.1` 的 `@aether/settings` 过时描述已清理。相对本节初稿的两处偏差：① `kb_manage` 的 `create`/`add_note` 也要确认（不再是「纯新增免确认」）；② `refresh` 定义保留但实现推迟到 P3（已于 PR #496 落地；P3 另增 `add_url`/`add_workspace`/`retry_embeddings` action）。单测见 `test/shared/mcp_tools/knowledge/knowledge_tools_test.dart`。

### 7.1 与 `@aether/settings` 的命名冲突（需现在拍板）

`builtin_mcp_servers.dart` 里 `@aether/settings` 的描述写着"让 AI 管理知识库（创建、编辑、删除、搜索）"，但 `builtin_tool_catalog.dart` 里它实际的工具列表**只有 provider/model 管理**，没有任何知识库工具——这是 web 端遗留的过时文案，从未清理。

## 8. 独创点：workspace 作为知识源

`knowledge_item.type == workspace` 时，`source` 存一个 `Workspace.id`，摄取阶段通过 `WorkspaceBackend.listDir` / `readFile` 遍历该工作区目录建索引。

这把项目里两个"双轨"领域打通：工作区负责"读写文件"，知识库负责"对同一批文件做语义检索"——用户打开的项目目录，不用手动逐个添加文件，一键就能被聊天语义搜索到。技术上零风险：只是多了一种 `knowledge_item` 来源，摄取/检索/工具三层完全不用感知来源差异。

### 8.1 staleness 检测（而非只是风险提示）

SAF 只能感知"通过本 backend 的改动"，外部改动（另一个 App 改了同一目录）无法主动感知。给出具体机制而非停留在"需要提示"：

- 摄取 `type: workspace` 的 `knowledge_item` 时，额外记录每个源文件的 `(path, mtime, size)` 快照到 `knowledge_item.sourceFingerprint`（JSON）。
- `kb_search` 命中后，**不在检索路径上做同步 IO 校验**（避免拖慢检索）；而是命中展示时后台异步用 `WorkspaceBackend.listDir`/`getFileInfo` 比对指纹，不一致则仅在该条引用上标一个 `possiblyStale: true`，UI/工具结果里带一句"来源可能已变更，建议重新索引"，不拦截、不影响本次回答。
- 真正的重新拉取仍然是显式的 `kb_manage refresh`，staleness 检测只负责"提醒"，不自动触发（避免在用户不知情时消耗 embedding 额度）。

## 11. 需要尽早决策的风险点

- **移动端 sqlite-vec 可用性**：`SqliteVecService.probe()` 在部分设备/架构上可能失败，必须保证 Dart 余弦全链路无缝兜底，且性能在几千 chunk 量级下可接受。
- **切块算法的移植成本**：Cherry Studio 的结构感知切块（markdown 断点评分 + 代码围栏保护）价值高但实现量不小，P0 可先用简单定长切块，P1 再升级，不阻塞骨架落地。
- **Embedding 成本可见性**：需在 UI 上明确提示"重新索引未变内容不重复扣费"，管理好用户预期（同记忆功能的哈希去重口径）。
- **workspace 源的一致性**：见 §8.1，已给出具体的 staleness 检测机制，不再是一条悬空的风险提示。

### 11.1 存储配额与孤儿数据 GC

手机存储有限，长文档 + 向量 blob（`kb_embedding.vectorBlob`）容易失控。P3 前至少要有：

- 单库/全局大小的软上限提示（建库/加文档时提示预估占用）。
- `knowledge_item` 被删除后，`kb_chunk` / `kb_embedding` / `kb_vec_<dim>` 里对应的行必须级联清理；`kb_embedding` 因为按 `embeddingKey` 去重、可能被多个 chunk 共享，删除前要确认没有其它 chunk 仍引用该 key（引用计数或"删除后跑一次孤儿扫描"二选一，后者更简单，可放进定期维护任务）。

### 11.2 备份策略（当前系统已隐含一个错误预期）

> **实现状态（P3d 已落地，PR #499）**：下述决策已按本节方案实现——权威三表进备份（`knowledge.json`），派生数据不进备份、恢复后重建切块（向量走 `retry_embeddings` 补嵌）；`kb_embedding` 可选进备份未做。

`features/backup/data/backup_service.dart` 此前把 `knowledgeBases` / `knowledge` 列为**恢复时必定丢弃、不支持**的类别——这条判断是"旧版 web 备份格式兼容性对不上"产生的，并非"本地知识库不需要备份"的结论，知识库真正落地后必须重新决策，不能让这条遗留逻辑继续生效：

- **权威数据**（`knowledge_base` / `knowledge_item` / `knowledge_content`）应纳入本地备份——否则换机/重装即全部丢失，且这部分数据量可控（正文文本，非向量）。
- **派生数据**（`kb_chunk` / `kb_embedding` / `kb_vec_<dim>`）体积可能达到几十 MB（向量 blob），建议**不进备份**，改为"恢复后自动触发全量重索引"——直接复用 Cherry Studio"派生数据可重建、不必持久备份"的原则，备份体积可以砍掉九成以上，代价是恢复后首次需要重新跑一遍 embedding（对齐 §5 的惰性去重，若备份保留了 `knowledge_content`，理论上也可以把 `kb_embedding` 一并可选备份供"不想重新付费"的用户勾选）。
- **workspace 源的知识库**尤其不该重复备份原文——`knowledge_item.type == workspace` 时正文已经是工作区文件的快照，工作区本身若有独立备份/同步策略，这里可以只备份 `knowledge_item` 的引用关系（source = workspaceId + 相对路径），不重复存一份文本。

## 12. 一句话总结

> 知识库不是"聊天的附加功能"，是和工作区同构的独立领域：
> 用一个 `KnowledgeScope` 统一"聊天可见 / 智能体可见"，核心摄取与检索逻辑只写一份；
> 复用记忆功能已验证的 Embedding + sqlite-vec 基建，复用工作区已验证的"Store + 内置 MCP 工具"双轨落地方式；
> 先打通聊天轨道，智能体轨道等其领域设计完成后即插即用。
