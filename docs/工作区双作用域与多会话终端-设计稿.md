# 工作区双作用域（项目模式 / 全机模式）与多会话终端 — 设计稿

> 状态：设计稿（未实施）。
> 前置阅读：《工作区与智能体模式-设计构想》（早期初稿，架构方向仍有效，进度信息已过时）、
> 《内置终端PRoot-设计文档》、《SSH工作区后端-设计文档》。
> 目标读者：Aetherlink 开发者。
>
> 结论先行：**同一个执行后端，两种作用域。** 工作区增加 `scope` 维度——
> 「项目模式」把智能体锚定在一个项目目录内（IDE 式真工作区），
> 「全机模式」保留完整终端能力（现状行为）。二者共享后端 / 会话池 /
> 工具层，只在「默认 cwd、审批策略、提示词注入」三处分叉。

---

## 0. 现状盘点（截至本稿）

早期初稿写作时只有 SAF 只读浏览；现在能力层已大幅落地：

| 模块 | 现状 |
|---|---|
| 后端 | 四种已落地：`localSaf` / `termux` / `ssh` / `prootLocal`（`WorkspaceBackendType`），统一走 `WorkspaceBackend` 抽象 |
| 终端 | 内置终端（PRoot + Alpine）、SSH、Termux 均 `canExec`；交互式终端页可用 |
| AI 工具 | 内置 MCP server `@aether/terminal`：`terminal_execute`（一次性）+ `terminal_session_*`（长驻会话池，超时后台续跑 + `tailOutput` 回看） |
| 会话池 | `workspace_session_pool.dart`：按后端池化，空闲 10 min 释放，单后端上限 4 会话 |
| 安全 | `terminal_command_guard.dart` 命令黑名单 + 聊天层 HITL 审批（执行类工具全量确认） |
| 文件工具 | `@aether/file-editor` 已按 `workspace.root` 解析路径（天然目录级隔离） |

**当前问题**：有终端的工作区（SSH / Termux / 内置终端）实际是「终端即工作区」——
`prootLocal` 的默认 cwd 是 `/root`，AI 拿到 `terminal_execute` 等于拿到整个
rootfs / 远程机器。没有「项目目录」这个边界，导致：

1. 审批压力全在用户：任何命令都要 HITL，无法按「目录内 / 目录外」分级放行；
2. 多项目无法并存：同一台 SSH 主机 / 同一个 rootfs 上开两个项目，会话、
   文件树混在一起；
3. 与 SAF 模型不一致：SAF 天然是「目录即工作区」，其余三种却是「整机即工作区」，
   智能体侧要写两套心智模型。

## 1. 目标

1. **项目模式（IDE 式真工作区）**：工作区 = 一个项目目录；文件工具、终端会话、
   未来的 stdio MCP 全部锚定在该目录内；目录外操作升级审批。
2. **全机模式（完整终端）**：保留现状——工作区 = 整个执行环境（rootfs 根 /
   SSH `$HOME`），全部命令走现有黑名单 + 全量 HITL。
3. **多会话终端**：一个工作区可同时开多个终端 tab（用户侧）/ 多个池化会话
   （AI 侧），互不干扰；不同工作区的会话彼此不可见。
4. **环境隔离分层**：默认目录级软隔离；按需叠加语言级虚拟环境隔离；
   强隔离（独立 rootfs）仅作远期可选项。

非目标：容器级强隔离（手机上一个 rootfs 数百 MB～GB，不值得默认做）；
桌面端 UI；agent 领域层本身（那是初稿阶段 B，另行设计）。

## 2. 领域模型改动

### 2.1 `Workspace` 增加 `scope`

```dart
/// 工作区的作用域：决定默认 cwd、审批策略与提示词注入。
enum WorkspaceScope {
  /// 项目模式：root = 项目目录，工具被锚定在 root 内（IDE 式）。
  project,

  /// 全机模式：root = 执行环境的根（rootfs `/` 或 SSH `$HOME`），
  /// 等价于现状「终端即工作区」。
  full,
}

class Workspace {
  // ...现有字段...
  final WorkspaceScope scope; // JSON 缺省 → 按 root 推断（见 2.3）
}
```

- `localSaf` 永远是 `project`（SAF 授权本身就是目录级）。
- `ssh` / `termux` / `prootLocal` 两种都可选，创建入口区分（见 §5）。

### 2.2 各类型的 root 约定

| backendType | project 模式 root | full 模式 root |
|---|---|---|
| localSaf | `content://` treeUri（现状） | 不适用 |
| prootLocal | rootfs 内 `/root/projects/<name>`（创建时 `mkdir -p`） | `/`（cwd 默认 `/root`，现状） |
| ssh | 远端某目录（现状已是工作区根） | `$HOME` |
| termux | Termux 内某目录 | `$HOME`（Termux prefix 下） |

### 2.3 兼容迁移

- 旧记录无 `scope` 字段：`localSaf` → `project`；其余按 root 推断——
  root 为空 / `/` / `$HOME` → `full`，否则 → `project`。
- 无需数据迁移脚本，`Workspace.fromJson` 里兜底即可（与 `connectionId`
  的 back-compat 处理同一风格）。

## 3. 执行层改动

### 3.1 会话锚定工作区

现状：`@aether/terminal` 的 `_resolveTarget` 不传 `workspace` 时落到
内置终端、cwd `null → /root`；会话池按「后端」池化，`workspaceLabel`
只是展示名。

改为**按「工作区」锚定**：

1. `_resolveTarget` 返回值带上 `Workspace`（含 scope / root）；
   内置终端默认目标改为「当前工作区」而非裸后端。
2. 会话池键从「后端」细化为「后端 + workspace.id」：不同工作区各自的
   会话列表隔离，`terminal_session_list` 默认只列当前工作区的会话
   （可加 `all: true` 参数查看全部）。
3. session 创建时强制 `cwd = workspace.root`（project 模式）并注入
   环境变量：`WORKSPACE_ROOT`、`WORKSPACE_NAME`。full 模式维持现状。

### 3.2 命令守卫升级：黑名单 → 黑名单 + 作用域规则

`terminal_command_guard.dart` 保留现有黑名单（两种模式都过），
project 模式额外叠加**路径越界检测**：

```dart
/// project 模式下的命令风险评级。
enum CommandRisk {
  /// root 内、命中低危白名单（ls/cat/git status/build 等）→ 免审批直接执行。
  safeInRoot,

  /// root 内、不在白名单 → 走现有 HITL（默认行为，兜底）。
  needsApproval,

  /// 引用了 root 之外的绝对路径 / `cd` 出 root / 提权类命令 → 强 HITL，
  /// 审批弹窗高亮越界原因。
  escapesRoot,
}
```

- 越界检测是**启发式**（解析命令里的绝对路径、`~`、`..` 穿越、`cd` 目标），
  不追求完备——它只决定审批等级，不是安全边界本身；真正的兜底仍是
  HITL + 黑名单 + rootfs 沙箱。
- 低危白名单从小做起（只读类 + 常见构建命令），宁缺勿滥；用户可在
  设置里关掉「root 内免审批」回到全量确认。
- full 模式不引入白名单放行——完整终端的代价就是全量审批，这是
  两种模式的核心差异，也是给用户的心智锚点。

### 3.3 交互式终端页

- 工作区终端页（`workspace_terminal_page.dart`）支持**多 tab**：
  每 tab 一个独立 PTY 会话，生命周期由页面管理（与 AI 会话池分开，现状纪律不变）。
- project 模式下新开 tab 的初始 cwd = `workspace.root`；full 模式 = 现状。
- tab 标题默认 `1 / 2 / 3`，可重命名；会话在后台保活直到页面销毁或手动关闭。

### 3.4 AI 会话缺口：向运行中进程写 stdin

现有 `terminal_session_*` 缺「往运行中命令写输入」的能力（跑交互式
程序、回答 y/n 提示所必需）。补一个工具：

```
terminal_session_write(session_id, input, append_newline = true)
```

底层即 `WorkspaceShellSession` 的 stdin 写入（PTY 已具备）。该工具属
执行类，纳入 `terminalToolNeedsConfirmation`。

## 4. 环境隔离分层

| 层级 | 手段 | 默认？ |
|---|---|---|
| L1 目录级（软隔离） | scope=project + cwd 锚定 + 越界审批 | ✅ 项目模式默认 |
| L2 语言级 | 每工作区独立 venv / 局部 node_modules；可选每工作区独立 `HOME`（`/root/projects/<name>/.home`，注入 `HOME` 环境变量，隔离 rc 文件与全局配置） | 可选开关 |
| L3 rootfs 级（强隔离） | 每工作区独立 PRoot rootfs 实例 | ❌ 远期可选，默认不做 |

设计取舍：共享 rootfs 的系统包（node / python / git）是**优点**——
装一次全工作区可用；隔离粒度放在项目产物（依赖目录、虚拟环境、HOME）
这一层，成本低、覆盖 IDE 实际需要的 95% 场景。这也是 VSCode / Cursor
在非 devcontainer 场景的通行做法；Operit 的 Ubuntu 沙箱同样是
「共享环境 + 目录即工作区」。

## 5. UI / 交互

创建工作区入口按「先选后端，再选作用域」组织：

```
打开工作区
├─ 本地文件夹 (SAF)            → 只有项目模式
├─ 内置终端 (PRoot)
│   ├─ 打开项目文件夹…         → project（在 /root/projects/ 下选/建目录）
│   └─ 打开完整终端            → full（root=/，即现状）
├─ SSH
│   ├─ 打开远端目录…           → project（现状 SSH 工作区即此形态）
│   └─ 打开完整终端            → full（root=$HOME）
└─ Termux（同 SSH 结构）
```

- 工作区列表 / 标题栏用徽标区分模式（如「📁 项目」/「🖥 全机」），
  让用户随时知道当前 agent 的权限边界。
- 审批弹窗在 project 模式下展示相对 root 的路径与越界高亮。

## 6. 提示词注入（agent 侧）

按 scope 注入不同的系统提示上下文：

- **project**：「你的工作区根目录是 `<root>`（工作区名 <name>）。所有文件
  操作与命令默认在该目录内进行；访问目录外路径需要用户确认。」
- **full**：「你拥有 <label> 的完整终端权限（root=<root>）。所有命令执行
  都需要用户确认，谨慎操作。」

这与 `@aether/file-editor` 现有的 workspace 解析对齐，agent 无需新心智。

## 7. 与 stdio MCP 的衔接（前瞻，另立设计稿细化）

现状 `remote/` 仅支持 SSE / Streamable HTTP。有了项目模式后，
stdio MCP server 的运行形态自然确定：

- 启动目录 = `workspace.root`，进程挂在该工作区的会话池上（生命周期、
  空闲回收复用 §3.1 的机制）；
- 内置终端里经 PRoot Alpine 的 node / python（npx / uvx）拉起；SSH
  工作区在远端拉起（等价 `ssh host npx …`）；
- transport 层新增 `StdioTransport`：JSON-RPC 按行分帧接到现有 MCP 协议层。

本稿只锁定「stdio MCP 按工作区锚定」这一结论，实现细节另写。

## 8. 落地顺序

1. **P0 领域**：`WorkspaceScope` 枚举 + `Workspace` 序列化兼容 +
   创建入口 UI（内置终端先行：项目文件夹 / 完整终端两个入口）。
2. **P1 执行**：会话池按 workspace.id 隔离；session cwd/env 锚定 root；
   `terminal_session_list` 按工作区过滤。
3. **P2 守卫**：`CommandRisk` 三级评级 + 越界启发式 + 低危白名单
   （带设置开关）；审批弹窗展示越界原因。
4. **P3 终端页**：多 tab 交互式终端。
5. **P4 补齐**：`terminal_session_write`（stdin 写入）。
6. **P5 可选**：L2 语言级隔离开关（独立 HOME / venv 约定）。
7. **远期**：stdio MCP（独立设计稿）、L3 独立 rootfs、SSH/Termux 的
   full 模式入口。

每步独立可合并、可单测（守卫与 scope 推断均为纯 Dart，测试放
`test/features/workspace/` 与 `test/shared/mcp_tools/`）。

## 9. 风险与开放问题

- **越界启发式的误报/漏报**：定位为「审批分级器」而非安全边界，漏报兜底
  是 HITL + 黑名单 + 沙箱；误报只是多弹一次确认，可接受。
- **会话池按工作区细分后的资源上限**：`kMaxPooledSessions` 从「每后端 4」
  改为「每工作区 4 + 全局上限（如 8）」，防多工作区并发耗尽内存。
- **full 模式与项目模式同后端并存**：同一 rootfs 上 full 工作区可以看到
  project 工作区的目录——这是共享 rootfs 的固有属性，通过 UI 徽标 +
  提示词声明来管理预期，不做技术阻止。
- **SAF 与 exec 的组合**：SAF 工作区永远 `canExec=false`；若用户想给
  SAF 目录加终端能力，引导其改用内置终端工作区（把项目放进 rootfs）。

---

## 10. 一句话总结

> 后端只有一份，作用域分两种：项目模式给智能体一个有边界的「手术台」，
> 全机模式保留完整终端的自由度；会话池、守卫、提示词按 scope 分叉，
> 其余全部复用——这是通往 IDE 式真工作区与 stdio MCP 的最短路径。
